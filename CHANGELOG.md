- ci: DAST ограничивает время работы ZAP и при любом завершении удаляет только свой одноразовый контейнер. Зависший сканер больше не удерживает E2E resource group и не блокирует следующие pipeline.

- ci: nightly L4/L7 проверяет, что адаптивный `DROP` и правило `WAF-RUNTIME-L4` исчезают после quiet decay; ускоренные параметры применяются только к одноразовому E2E-контуру.

- ci: E2E-проверка порядка AntiBot и активного rate-limit ожидает завершения переключения worker-процессов Nginx после apply revision. Это исключает ложный `404` default vhost во время reload, не ослабляя обязательный итоговый `429` для подтверждённого клиента.

- ci: исправлена передача пути к E2E compose-файлу в shell-runner, из-за которой smoke-проверка завершалась до запуска тестов.

### Дашборд и E2E

- ci: E2E-runner больше не запускает слепой initial compile/apply поверх dev-fast-start revision: это исключает гонку, при которой runtime reload listener останавливался до первого проверяемого изменения политики.
- ci: E2E-проверка исключения management host из L7 rate-limit вынесена в отдельный чистый job, чтобы изменение runtime в других security-сценариях не искажало её результат.
- ci: добавлены nightly E2E-проверки восстановления runtime после restart и краткой недоступности control-plane, Vault и Postgres; активная revision продолжает блокировать атаки.
- security: добавлены отрицательные DAST-пробы SQLi, XSS и path traversal, контроль анонимного administrative API и проверка отсутствия ложной блокировки легитимного запроса.
- ci: доказательные E2E-артефакты содержат безопасные шаги и HTTP-наблюдения по тестам; диагностические логи перед публикацией редактируют password, Authorization, cookie, token и secret.
- ci: полный E2E запускает только проверяемые шаги шаблонов AntiBot, а полный challenge-flow остаётся обязательной smoke-проверкой; намеренный `skip` больше не может скрыть пропущенное доказательство.
- ci: JSON и Markdown-артефакты E2E дополняются checksum активной runtime-конфигурации, revision ID, HTTP-статусами, причинами блокировок, а для L4/L7 — снимками adaptive entries и правил `WAF-RUNTIME-L4`.
- ci: E2E evidence JSON теперь включает revision ID, наблюдаемые HTTP-статусы, причины блокировки и checksum активной runtime-конфигурации.
- ci: E2E разделён на обязательный smoke и полный набор после smoke; автоматический retry удалён, чтобы flaky-падение оставалось ошибкой.
- ci: полный E2E создаёт временный `.env` auto-start из versioned-шаблона, как штатный инсталлятор, и удаляет его после проверки.
- core: удалённые или выключенные сайты больше не передают доверенные proxy CIDR в активную конфигурацию runtime.

- Исправлена агрегация защищённых сайтов на дашборде: путь `/` больше не считается административным вне management-host. Блокировки AntiBot со статусом `403` учитываются в 24-часовых итогах, графике и топе IP атакующих.
- E2E подтверждает полный путь: восемь блокировок AntiBot попадают в «Запросы» с причиной `antibot`, затем в runtime-агрегат и дашборд с точным IP атакующего.
- L4/L7 E2E использует отдельный контейнер атакующего, применяет защитный revision последним и проверяет цепочку `429` → adaptive `DROP` → правило iptables → блокировка новой TCP-сессии без зависимости от порядка других сценариев.

### Безопасность Basic Auth

- Проверка Basic Auth завершается после выполнения `auth_basic` Nginx, поэтому неверные, отключённые и межсайтовые учётные данные отклоняются.
- Отключённые пользователи больше не активируются устаревшими полями профиля, а сгенерированные файлы учётных данных изолированы по сайтам.
- E2E ждёт обновлённую session-cookie из активной ревизии runtime перед проверкой отзыва сессии после смены пароля.

### Публикация и CI/CD

- E2E-runner теперь считает любой `skip` ошибкой доказательной проверки; необходимые runtime endpoint’ы и feature flags передаются явно, Basic Auth ожидает активации защищаемого виртуального хоста, а AntiBot-поток проверяется на скомпилированном TLS-сайте без пропуска.
- Добавлен изолированный DAST-контур OWASP ZAP после security-гейтов: baseline проверяет `main` и теги, а активный full scan запускается по расписанию и для release-тегов. Отчёты ZAP и двуязычный DAST evidence прикладываются к GitHub Release; High/Critical блокируют pipeline.
- Ручные jobs публикации в GitHub и GitHub Release жёстко зависят от успеха validate, unit, E2E, security и DAST; при ошибке внешняя публикация недоступна.
- GitLab CI разделён на быстрый E2E smoke-набор, полный E2E-набор и отдельный набор security-инвариантов: Basic Auth, межсайтовая изоляция, compile/apply, антибот, ModSecurity, rollback и security-телеметрия.
- E2E подтверждает, что события неудачной Basic Auth-аутентификации сохраняются с причиной `auth`, но пароль и значение заголовка Authorization не попадают в телеметрию запросов.
- Каждый E2E job формирует двуязычный отчёт-доказательство (`e2e-evidence.md` и `e2e-evidence.json`) без секретов; при публикации релиза эти отчёты прикладываются к GitHub Release.
- Перед публикацией Docker-образа релиза CI проверяет сгенерированные Nginx-артефакты, подпись, SBOM и provenance, затем запускает собранный образ и ждёт `/healthz`.
- После успешного GitLab pipeline для ветки `main` доступна ручная job зеркалирования в GitHub.
- Ручная job зеркалирования GitHub больше не допускает завершение с ошибкой: при запуске она блокирует pipeline до успешной публикации либо честно отмечает его неуспешным.
- После успешного pipeline тега версии `vX.Y.Z` доступна ручная job публикации подписанных метаданных релиза, GitHub Release и Docker-образов версии и `latest` в GHCR.
- В release-скрипте полный набор локальных проверок запускается только в режиме `full`; отдельный режим GitLab публикует чистую ветку `main` без ожидания E2E.
- GitLab CI проверяет репозиторий, полный набор Go-тестов, сборку документации, Docker E2E для штатного и чистого onboarding-сценариев, Gitleaks, Trivy, govulncheck и npm audit.
- E2E-исполнитель Windows надёжно сохраняет вывод Docker BuildKit и определяет результат по коду завершения Docker.

### Наблюдаемость и панель управления

- Исправлена кнопка показа пароля Basic Auth: сохранённый пароль раскрывается отдельным авторизованным и аудируемым запросом, а маска показывает фактическую длину пароля.
- В «Запросах» добавлен локализованный фильтр причины блокировки. Ошибки аутентификации и антибота записываются с точными причинами `auth` и `antibot`; успешная аутентификация остаётся обычным запросом.
- Из пользовательского интерфейса «Запросов» убраны краткая security-сводка, служебные legacy-поля и `legacy_row_type_support`; совместимость определяется только на сервере.
- Средняя отметка оси графика запросов и атак за 24 часа теперь равна половине пика: при пике 54 показывается 27.
- Виджеты запросов дашборда используют единый защищённый агрегат за полное окно 24 часа. Итоги, почасовые ряды, топ сайтов и URL, уникальные IP, ошибки и security-метрики больше не зависят от ограничения пагинации списка запросов.
- Добавлен защищённый `GET /requests/dashboard-summary`; он исключает трафик панели так же, как дашборд, но сохраняет security-события защищаемых сервисов.
- Успешные блокировки ModSecurity не классифицируются как ошибки состояния контейнеров и остаются доступны в security-телеметрии.
- Топ IP атакующих содержит наблюдаемую страну, поэтому строки дашборда и переходы к деталям не теряют контекст.

### Страницы аутентификации и ошибок

- Встроенные endpoint’ы проверки Basic Auth и service token больше не блокируются фильтром разрешённых методов сайта; POST-проверка входа работает при любой пользовательской политике методов.
- Добавлены девять выбираемых брендированных вариантов Basic Auth с защищёнными предпросмотрами, сохранением настройки, серверной валидацией, генерацией runtime-артефактов и E2E-проверкой; варианты, которые ранее были `v5`–`v9`, теперь занимают позиции `v1`–`v5`.
- Страницы Basic Auth локализованы для `ru`, `en`, `de`, `sr` и `zh`; улучшены логотипы, контраст, выравнивание и расположение управляющих элементов предпросмотра.
- Селекторы вариантов Basic Auth и антибота выровнены по высоте, а варианты Basic Auth названы нейтрально: «1 вариант», «2 вариант» и далее.
- Скорректированы светлая синяя тема `v4`, текст `v6`, а также логотипы и компоновка вариантов `v7`–`v9`.
- Кастомные страницы ошибок получают Request ID, IP клиента и время из `Server-Timing` после внутреннего `error_page`-редиректа; эти значения больше не отображаются как «н/д».

### Первичная настройка и хост управления

- Первичная настройка сохраняет домен панели в management hosts до первой компиляции ревизии, поэтому CRS не блокирует control-plane API.
- Запросы с кодами 403 и 444 к management host сохраняются как security-события, тогда как обычный UI/API-трафик панели исключается.
- Глобальные L7-лимиты Anti-DDoS исключают все явно настроенные management hosts, включая установки с произвольным идентификатором сервиса.
- Панель управления и onboarding не подменяют DOM устаревшей синтетической страницей 429 после фонового API-запроса или ошибки статики.
- Свежий HTTP-onboarding использует отдельную пару session-cookie и корректно переходит на обычную Secure-сессию после HTTPS-входа.
- E2E-onboarding проверяет bootstrap, выпуск и привязку self-signed TLS, применение ревизии и вход по HTTPS.

### Безопасность и зависимости

- Обновлена транзитивная зависимость `websocket-driver` до 0.7.5 для устранения CVE-2026-54466 / CWE-130 и отклонения некорректных oversized WebSocket length headers.
- Экспорт приватного материала сертификатов защищён свежей TOTP-проверкой, одноразовым подтверждением другим пользователем и атомарной фиксацией ошибок.
- Обновления правил WAF проверяют источник и SHA-256 точного архива до атомарной активации.
- Хранилища в корпоративных и лабораторных профилях требуют TLS, отдельные учётные данные и ограниченные сетевые политики; существующие тома, ключи, сертификаты и хэши не перезаписываются.
- Секреты, токены и параметры процессов не выдаются через статические nginx-пути, диагностику или экраны мониторинга.
- Ввод для nginx, ModSecurity, путей, идентификаторов, upstream-сервисов и доверенных прокси валидируется до генерации конфигурации.

### Проверки

- E2E Basic Auth больше не зависит от bootstrap-сайта: каждый сценарий создаёт собственный сайт и upstream, а разрешённые HTTP-методы сохраняются в фактически используемом поле профиля.
- Добавлены E2E-проверки полного пути блокировки Basic Auth/антибота до «Запросов» и дашборда, согласованности runtime-агрегата с дашбордом при разных pagination и часовых поясах, отката отклонённой конфигурации, жизненного цикла и межсайтовой изоляции Basic Auth, а также доверенных прокси и защиты от подмены `X-Forwarded-For`.
- Добавлены регрессии для TOTP, экспорта сертификатов, CRS, ACME HTTP-01, прямого IP-доступа, виртуальных патчей, порядка антибота и rate-limit, телеметрии Basic Auth/антибота и метаданных кастомных страниц ошибок.
- Ручные проверки UI и двусторонних API/runtime-сценариев зафиксированы в `.work/UI_CHECKLIST.md`.
