## [1.2.2] - 17.05.2026

### Ядро
- Восстановлен корректный экспорт `ensureControlPlaneAccessManagementMethods` в модуле сервисов, из-за которого не загружался runtime-бандл страницы сервисов.
- В шаблонах runtime NGINX скорректирован `Server-Timing`: теперь `rid/ip/ts` отдаются из `$request_id/$remote_addr/$msec`, чтобы fallback-страницы 4xx/5xx показывали реальные значения метаданных.
- Исправлен жизненный цикл auth-сессий при рестарте control-plane: boot-токен сессии больше не фиксируется на `pepper`, поэтому после перезапуска активные браузерные сессии корректно инвалидируются и требуется повторный вход.
- В site-шаблоне runtime NGINX добавлен выделенный `location ^~ /static/` без site-rate-limit include, чтобы загрузка UI-модулей (`/static/js/pages/*.js`) не срывалась `429` на странице сервисов.
- Для локального compose-профиля повышен дефолт `WAF_DEFAULT_LIMIT_REQ_RATE` до `120r/s`, чтобы штатная загрузка SPA-модулей и healthcheck не попадали под ложные `429` на `localhost`.
- Для динамических импортов страниц в `app.js` включён cache-busting (`?v=...`), чтобы браузер не держал старые сломанные ESM-модули после hotfix и перезапуска контейнеров.
- Реализована строгая семантика `security_mode` для easy-профилей сервисов:
  - `transparent` — полностью пассивный режим (0 защитных модулей, без enforcement),
  - `monitor` — только наблюдение/логи (без блокирующих модулей),
  - `block` — полное применение всех включенных модулей.
- Политика `security_mode` теперь применяется не только в основном рендере easy-конфига, но и в генерации custom rate-limit артефактов, чтобы в `transparent/monitor` не создавались блокирующие `limit_req` правила.
- Применение `security_mode` перенесено в серверную нормализацию easy-профиля (`control-plane`): сохранённый профиль и legacy-policy sync теперь всегда соответствуют реальной семантике режимов (`transparent/monitor/block`), а не только этапу компиляции.
- Обновлён `scripts/release.ps1`: добавлена публикация GitHub Release (`gh release create/edit`) с заметками из секции `CHANGELOG.md` для текущей версии.
- В `scripts/release.ps1` добавлен явный режим `publish` (`6`) для связки `release + docker` без ручного запуска отдельных шагов.
- Docker-публикация в release-скрипте использует версию из `control-plane/internal/appmeta/meta.go` (теги `tarinio:<version>`, `ghcr.io/berkutsolutions/tarinio:<version>`, `latest`), что корректно подхватывает `1.2.2`.

### Рефакторинг
- Исправлены ошибки декомпозиции в админке: возвращены именованные экспорты `renderUsersTable` и `renderRolesTable` из `administration.helpers-base.js`.
- Устранена ошибка импорта `renderRolesTable` в `administration.js`, вызывавшая падение вкладки администрирования при загрузке.
- Исправлена регрессия в `sites.detail-render-view.js`: удалён дублированный блок с повторным объявлением `SETTINGS_SEARCH_INDEX`, восстановлен корректный `return` шаблона.
- Исправлен синтаксис в `sites.detail-render-view-part2.js` (ошибка `Unexpected end of input`): восстановлен корректный шаблонный `return`, из-за которого ранее падал динамический импорт `sites.js`.
- Устранена финальная причина падения вкладки сервисов: в `sites.detail-render-view.js` был обрезан хвост шаблонной строки (после `renderDetailViewRuntimeTail(...)`), из-за чего браузер стабильно получал `SyntaxError: Unexpected end of input` на `/services`.
- Для стабилизации `/services` в Firefox добавлен временный compatibility-fallback detail-рендера (без ранней загрузки проблемного `sites.detail-render-view.js`) и облегчён preflight модулей сервисов.
- Восстановлена рабочая таблица сервисов на `/services`: загрузка секции переведена на стабильный `sites.js` c прямым чтением `/api/sites`, без зависимостей от проблемной цепочки `sites.page-main-*`.
- Для полного восстановления функционала (`list + edit + actions`) страница `/services` временно переключена на проверенный монолит `sites.js` из baseline `1.0.7`; декомпозированная цепочка `sites.page-main-*` сохранена в кодовой базе для отдельного этапа совместимости Firefox.

- Для точечной стилизации list-редакторов добавлены семантические hook-классы (`waf-list-editor`, `data-list-field`, `waf-http-max-size`, `waf-http-version-toggle`) без изменения бизнес-логики.

### Тесты
- Расширен e2e-smoke регресс (`TestE2EUIFullRegression`): добавлены проверки маршрутов `/services`, `/revisions`, `/profile` и жёсткая проверка загрузки всех page-модулей без `429`/HTML fallback.
- Добавлены и обновлены тесты компилятора для проверки фактического поведения режимов `transparent/monitor/block`:
  - `transparent` и `monitor` не выпускают блокирующие артефакты и не включают enforcement,
  - `block` сохраняет включенные защитные модули и лимиты.
- Добавлен e2e-тест `TestE2ESecurityModesReality`, который прогоняет все 3 режима (`transparent/monitor/block`) через API easy-профиля и проверяет фактические флаги защиты и состояние legacy-политик (`waf/access/rate-limit`).
- В `scripts/release.ps1` этот e2e-чек включён в обязательный `Startup Test Checks` (`go test ./ui/tests -run TestE2ESecurityModesReality -count=1`).
