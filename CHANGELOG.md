# Журнал изменений

Все значимые изменения проекта фиксируются в этом файле.

## [2.0.0] - 2026-04-19

### Security / Runtime hardening
- Аутентификация переведена на live-authorize модель: активные сессии теперь каждый раз проверяют текущее состояние пользователя и его актуальные роли, поэтому отключение пользователя или изменение ролей применяется без ожидания истечения session TTL.
- Смена пароля теперь инвалидирует все остальные сессии пользователя, а административный password reset также сбрасывает существующие сессии целевого пользователя.
- Добавлена прозрачная миграция password hash-ов на `Argon2id`: старые учетные записи продолжают входить без ручной миграции, но при успешной проверке пароля автоматически переводятся на новый формат.
- Усилен выпуск auth-cookie за reverse proxy: `Secure` теперь корректно включается не только при прямом TLS, но и при `X-Forwarded-Proto`/совместимых proxy-заголовках.
- Easy antibot больше не доверяет прямому bypass через `?waf_antibot=ok` и фиксированный cookie `waf_antibot=ok`; challenge переведен на site-scoped cookie и отдельную challenge-location, чтобы убрать тривиальный обход L7-защиты.
- L4 guard расширен на dual-stack модель: IPv4-цепочки сохраняют прежнее поведение, а IPv6 теперь получает отдельные правила и корректный `connlimit-mask 128`, где это поддерживается конфигурацией runtime.
- Политики доступа и rate-limit ужесточены для management-поверхностей: `control-plane-access` при наличии allowlist теперь компилируется в default-deny posture, а allowlist больше не выключает rate limiting полностью и получает отдельный rate-limit key вместо полного bypass.
- Runtime internal API (`/readyz`, `/reload`, `/security-events`, `/requests`, `/crs/*`) закрыт shared-token слоем через `WAF_RUNTIME_API_TOKEN` с совместимым loopback fallback, чтобы внутренние endpoints не оставались полностью trust-by-network.
- `GET /healthz` расширен до operational healthcheck: кроме revision store и catalog probe он теперь проверяет setup state, auth/session backend, zero-trust integrity пользователей и ролей, compiler preview без побочных эффектов, runtime readiness, security-events pipeline, requests pipeline и CRS status.
- Compose-окружения `default`, `auto-start` и `testpage` теперь автоматически прокидывают `WAF_RUNTIME_API_TOKEN` и не оставляют runtime probes в частично защищённом состоянии после обновления контейнеров.
- Встроенные роли (`admin`, `auditor`, `manager`, `soc`) теперь автоматически дозаполняются актуальными системными permission-ами при старте даже в старых volume, без ручной миграции `roles.json`.
- Container healthcheck control-plane сделан безопаснее для оркестрации: runtime probes переведены в мягкую деградацию с короткими таймаутами, чтобы `/healthz` не валил старт сервиса из-за временно недоступного runtime.
- Добавлены регрессионные тесты на live-permissions по сессии, автоматическую миграцию legacy password hash-ов, отзыв посторонних сессий при смене пароля и IPv6-ветку L4 guard.

### Zero Trust / Users / Roles
- Во вкладке `Администрирование` реализовано полноценное управление пользователями и ролями вместо прежних оболочек: списки, создание, просмотр по ЛКМ и редактирование через модальные окна.
- Добавлены базовые роли `auditor`, `manager`, `soc`, а также обновлённый набор системных permission-ов для zero-trust доступа по вкладкам и под-разделам.
- Дефолтная роль `admin` теперь принудительно получает полный набор разрешений, даже если role storage был изменён вручную.
- Разделён доступ по функциональным зонам: `Dashboard`, `Sites`, `Anti-DDoS`, `OWASP CRS`, `TLS`, `Requests`, `Events`, `Bans`, `Administration`, `Activity`, `Settings`, `Profile`.
- Для `Settings` введены отдельные права на `General`, `Storage` и `About`; обычным пользователям всегда доступны только свой `Profile`, `healthcheck` и `Settings -> About`, без автоматического доступа к `General` и `Storage`.
- Добавлены серверные API для управления учётными записями и ролями:
  - `GET/POST /api/administration/users`
  - `GET/PUT /api/administration/users/{id}`
  - `GET/POST /api/administration/roles`
  - `GET/PUT /api/administration/roles/{id}`
- Обновлены route guards: чувствительные endpoints теперь требуют не только общие CRUD-права, но и tab/module-level permissions для соблюдения zero-trust.

### Healthcheck / Tests / Docs
- В `healthcheck` добавлен отдельный zero-trust probe `GET /api/administration/zero-trust/health`, который проверяет stores пользователей и ролей, наличие обязательных ролей, полноту `admin` permissions и целостность ссылок user -> role.
- Обновлён compatibility contract healthcheck до `2026-04-19-healthcheck-v3` и синхронизирован с installer script.
- Добавлены обязательные тесты на модель ролей и прав с созданием временных тестовых пользователей в isolated test storage, без влияния на production-данные.
- Добавлены тесты на zero-trust health endpoint и на новые permission-профили ролей `auditor`, `manager`, `soc`.
- Актуализированы smoke- и UI-contract тесты под текущее состояние `2.0.0`: healthcheck больше не проверяется по устаревшему mojibake-маркеру, а dashboard smoke теперь валидирует реальные API-поля виджетов и актуальные DOM-маркеры.
- Добавлены backend-регрессии на dashboard service: тесты теперь отдельно проверяют корректное наполнение данных для виджетов, а также то, что статус `runtime` в карточках компонентов учитывает защищённый runtime probe вместо старого неавторизованного запроса.
- Актуализирована wiki-структура документации и changelog под новую подсистему пользователей, ролей и server-side zero-trust авторизации.

### Dashboard / UI
- Виджет статуса компонентов на dashboard переведён на авторизованный runtime probe, поэтому `runtime` больше не отображается как `DOWN` из-за отсутствующего runtime token в старом механизме проверки.
- Стартовые размеры виджетов `Компоненты UP`, `Компоненты DOWN`, `Количество запросов`, `Количество атак` и `Заблокированные атаки` увеличены для более ровной раскладки по умолчанию.
- Числовая типографика metric-виджетов и summary-блоков dashboard уменьшена, чтобы большие значения не выглядели перегруженно и лучше помещались в карточках.
- Runtime nginx получил более ёмкие proxy-буферы и отключение temp-file spill для proxy-ответов, чтобы не засорять healthcheck/логи предупреждениями вида `an upstream response is buffered to a temporary file` на статику UI и крупные API-ответы вроде `/api/requests`.
- Для default runtime server задан безопасный fallback `waf_site_id`, чтобы глобальные маршруты вроде `/static/*`, `/api/auth/login` и `/api/requests` не вызывали предупреждения `using uninitialized "waf_site_id" variable while logging request`.

### Ревизии / UI
- В боковое меню возвращена вкладка `Ревизии`.
- Раздел `Ревизии` переработан в сервисно-ориентированный экран: 80% ширины занимает фрейм с плитками сервисов, 20% — вертикальная колонка со статусами rollout и последними событиями по ревизиям.
- По ЛКМ на плитке сервиса открывается модальное окно с ревизиями этого сервиса, где доступны действия применения и удаления неактивных ревизий.
- Модальное окно ревизий сервиса переработано в аккордеон-плитки с более подробной информацией по каждой ревизии.
- Добавлены детальные модальные карточки статусов по ЛКМ на статус и очистка статусов ревизий.
- Очистка статусов перенесена во внутреннее статусное модальное окно, а сводные счётчики и лента событий синхронизированы с этой операцией.
- Глобальные стили карточек, заголовков и фреймов приведены к новому визуальному языку раздела `Ревизии` для остальных вкладок UI.
- Плитки ревизий в модальном окне приглушены по тону, активная ревизия закреплена в развёрнутом состоянии, а действия `Применить` и `Удалить` перенесены в заголовок рядом со статусом.
- Во фреймах страниц увеличен верхний внутренний отступ после разделителя заголовка, чтобы элементы не прилипали к линии на вкладках `TLS`, `Запросы`, `События`, `Баны`, `Администрирование`, `Активность`, `Настройки` и `Профиль`.
- Во вкладках `Anti-DDoS`, `OWASP CRS` и `Профиль` доработаны локальные визуальные стили, чтобы они лучше соответствовали новому общему языку интерфейса.
- Для `OWASP CRS` убран лишний внутренний фрейм статуса релиза, а блоки с текущим релизом и обновлениями вынесены в более чистый плоский layout.
- Во вкладке `Anti-DDoS` удалено устаревшее сообщение о ручной компиляции/применении, а секции модели и help-блоки лучше выровнены по отступам.
- Экран `Профиль` визуально сближен с административными и сервисными разделами: выровнены размеры кнопок и полей, плотность внутренних отступов и палитра фреймов.
- Во вкладке `Администрирование` исправлена таблица пользователей и ролей: action-кнопки больше не ломают нижнюю границу строк.
- Модальные окна пользователей и ролей переработаны визуально: permission-блоки сгруппированы по i18n-категориям, убраны режущие рамки и добавлена более аккуратная карточная сетка.
- В форме пользователя добавлены поля `Отдел` и `Должность`, отдельный placeholder для пароля при создании, подтверждение пароля и кнопки показа/скрытия пароля.

### Ревизии / Backend
- Добавлен API-каталог `GET /api/revisions` с агрегированными данными по сервисам, ревизиям и ленте статусов из snapshot/job/event storage.
- Добавлено удаление неактивных ревизий через `DELETE /api/revisions/{id}` с очисткой snapshot и связанных job-записей.
- Последнее применение ревизии теперь сохраняется в метаданных самой ревизии и не теряется после очистки статусной ленты.
- Добавлена очистка статусной ленты через API с реальной синхронизацией apply/error counters и истории событий.
- Исправлена классификация старых ревизий: ранее успешно применённые неактивные ревизии больше не отображаются как ожидающие применения.
- Расширена нормализация текстов статусов и событий ревизий, чтобы UI не показывал сырые строки вроде `apply succeeded` или `revision applied` без локализации.

### Healthcheck / Docs
- В `healthcheck` и в `GET /healthz` добавлена отдельная проверка API ревизий, чтобы деградацию revision catalog можно было отслеживать отдельно от базового revision store.
- Русская документация в `docs/ru` дополнена и русифицирована, включая отдельное подробное описание работы anti-DDoS модели и operator runbooks.
- Обновлены API-доки для каталога ревизий, статусов ревизий и healthcheck-проверок.
- Локализация `ru` и контроль качества i18n ужесточены: для русских строк добавлены дополнительные проверки на англоязычные фрагменты и расширен список допустимых технических терминов.
