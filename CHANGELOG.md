## [1.5.12] - 21.07.2026

### E2E и доказательства

- E2E-объекты получают уникальные идентификаторы: повторный запуск или расширение набора не может использовать ресурсы другого теста только из-за совпадения имени.
- Добавлен сценарий сохранения активной защиты: невалидный candidate не заменяет активную revision, а уже включённое правило ModSecurity продолжает возвращать `403`.
- В артефактах каждого E2E появился `e2e-stability.json`: в нём записаны набор, время запуска, число попыток теста, повторы сборки и признак инфраструктурной нестабильности. Функциональные тесты автоматически не повторяются.
- В GitHub Release evidence добавлены русские описания каждого E2E/DAST-набора и таблица Trivy для исходного дерева и публикуемого образа.

### Сессии и приватность

- Security-invariants проверяет, что успешный вход обновляет `last_login_at`, а смена пароля отзывает все остальные сессии пользователя, сохраняя текущую.
- Basic Auth E2E продолжает проверять, что пароль и значение `Authorization` не попадают в request telemetry; смена пароля отзывает выданную Basic Auth-сессию. Успешный Basic Auth вход теперь передаёт из runtime только site, имя пользователя и время через защищённый internal callback, поэтому поле «Последний вход» больше не остаётся «Никогда».

### Supply chain и CI/CD

- Перед release добавлена обязательная проверка Trivy конечного образа control-plane на High/Critical уязвимости. Исходный отчёт образа прикладывается к GitHub Release вместе с SBOM, подписью и provenance.

### Интерфейс и маршрутизация

- Страницы входа остаются скрытыми до получения выбранного оформления, а URL статических файлов оформления обновлён для исключения показа закешированного предыдущего интерфейса.
- Неизвестные пути UI больше не отдаются как SPA dashboard: известные deep-link маршруты сохранены, остальные пути возвращают `404`.

### Публикация и CI/CD

- Исправлена автоматическая публикация: release job получает DAST evidence до создания GitHub Release; версия повышена после уже опубликованных тегов `v1.5.6`, `v1.5.7` и `v1.5.8`.
- Извлечение заметок релиза перенесено из shell-команды в проверяемый Python-скрипт, устойчивый к окончаниям строк в `CHANGELOG.md`.
- В release job добавлены явные контрольные отметки для сбора evidence, публикации GHCR и GitHub Release, чтобы журнал точно показывал остановившийся шаг.
- Standalone health-check release-образа запускается с обязательным тестовым security pepper и явным loopback-портом, поэтому проверяет реальный `/healthz` до публикации в GHCR.
- Публикация GitHub Release и доказательств больше не зависит от внешнего образа GitHub CLI: она выполняется через GitHub REST API с тем же защищённым CI-токеном.

### Runtime и E2E

- Apply теперь подтверждает не только доступность runtime, но и загрузку именно целевой revision. При сохранении старой revision операция завершается ошибкой и запускает штатный rollback вместо ложного успешного статуса.

## [1.5.6] - 20.07.2026

### Исправления runtime

- Runtime ожидает завершения публикации файлов активной revision, если pointer появился раньше полного bundle. Это устраняет гонку запуска E2E и DAST.

### Сервисы и конфигурация

- Исправлена синхронизация простого и сырого редакторов сервиса. Любое изменение в простом режиме сразу отражается в `WAF_SITE_*`, а изменение в сыром режиме возвращается в простой draft и сохраняется в ресурсах сервиса.
- Изменение upstream, например `127.0.0.1:8080` на `privatebin:8080`, теперь проходит через API, скомпилированную ревизию и активную конфигурацию runtime без сохранения старого значения.
- Белые и чёрные списки больше не перезаписываются compatibility-bridge профиля Easy Site. Основная policy `<site>-access` имеет приоритет над технической `easy-<site>-access`.
- При сохранении из обоих вариантов редактора сначала сохраняется профиль Easy Site, затем каноническая access policy, после чего выполняется единый compile/apply.

### Защита и runtime

- Runtime ожидает фактической готовности HTTP-прокси на активной ревизии после перезагрузки Nginx. Это устраняет переходное окно между старыми и новыми worker-процессами.
- Launcher сохраняет защищённые health и reload endpoint при аварийном завершении контролируемого Nginx, что позволяет откатить применение на последнюю рабочую ревизию.
- Улучшена изоляция Basic Auth между сайтами, обработка graceful reload и проверка разрешённых HTTP-методов для endpoint проверки входа.
- Усилены проверки trusted proxy: подмена `X-Forwarded-For` контролируется для IPv4, IPv6, доверенных и недоверенных цепочек.
- E2E подтверждает реальные режимы защиты AntiBot, ModSecurity, rate limit, L4/L7 adaptive protection, rollback и отказоустойчивость runtime.

### Интерфейс и наблюдаемость

- Исправлено раскрытие сохранённого пароля Basic Auth: поле маскируется по фактической длине, а показ значения выполняется отдельным авторизованным и аудитируемым запросом.
- В «Запросах» добавлен фильтр причины блокировки. Блокировки Basic Auth и AntiBot получают причины `auth` и `antibot`; служебные legacy-поля исключены из интерфейса.
- Дашборд использует единый агрегат защищаемого трафика за 24 часа; средняя отметка шкалы равна половине пика.
- Добавлены девять локализованных вариантов страницы Basic Auth, улучшены предпросмотры, логотипы, контраст и выравнивание селекторов AntiBot и аутентификации.
- Кастомные страницы ошибок получают request ID, IP клиента и время через `Server-Timing`.

### E2E и доказательства

- Добавлен E2E-контракт реального защищаемого сервиса: ресурсы UI/API, candidate revision, active runtime и фактический HTTP-результат должны совпадать. Разрешённый IP получает `200`, неразрешённый — `403`.
- Добавлены контракты полного round-trip для всех полей simple draft и переключения `Простой → Сырой → Простой`; старое значение upstream не может остаться в одном из представлений.
- Каждый E2E job формирует очищенные JSON и Markdown доказательства с тестами, HTTP-статусами, revision ID, checksum конфигурации и причинами блокировок.
- При падении E2E сохраняются состояние Compose и диагностические логи runtime, control-plane и sentinel до teardown.
- Smoke, security-invariants, full и nightly разделены по назначению; пропущенные E2E не считаются доказательством исправности.

### DAST и безопасность зависимостей

- Canary-upstream DAST запускается от непривилегированного пользователя; исправлена находка Trivy `DS002`.
- В DAST enforcement добавлен безопасный API fuzzing: malformed JSON, лишние поля и oversized header обязаны завершаться контролируемым не-5xx ответом.
- Исправлена проверка DAST telemetry: сопоставление защищённого запроса учитывает query string, сохраняя обязательные проверки `403`, причины `modsecurity` и недоступности upstream.
- DAST выполняется в общем pipeline на `main` перед release stage. На период обкатки временно отключены остальные CI jobs и публикация, поэтому запускаются только `dast:baseline` и `dast:negative`.
- Исправлена найденная DAST-проверкой проблема management UI: добавлены `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy` и ограничивающая Content Security Policy.
- Добавлен отдельный ручной downstream pipeline DAST без ожидания unit и полного E2E. Основной pipeline временно не запускает тяжёлые DAST jobs; для version tag DAST остаётся release-gate, чтобы evidence был приложен к GitHub Release.
- DAST enforcement использует canary-upstream: вредоносный запрос обязан получить блокировку WAF, не попасть в приложение и появиться в запросах с причиной `modsecurity`; легитимный запрос обязан достичь canary.
- Добавлены authenticated DAST-контракты control-plane, проверка отзыва сессии, session cookies и безопасных response headers, а ZAP-результаты сопоставляются с версионируемой baseline policy.
- DAST запускает OWASP ZAP только против одноразового E2E-контура WAF и сохраняет JSON, HTML и Markdown отчёты. High и Critical находки блокируют релиз.
- Исправлены запуск ZAP с writable home, обработка Automation Framework и разбор числовых risk code в доказательстве DAST.
- Добавлены негативные security-пробы SQLi, XSS, path traversal, административных API без сессии и легитимных запросов.
- CI выполняет Gitleaks, Trivy, govulncheck и npm audit; обновлена transitive-зависимость `websocket-driver` для устранения CVE-2026-54466.

### Публикация и CI/CD

- После quality gates GitHub mirror и release/GHCR публикация выполняются автоматически по порядку: сначала mirror, затем release. CI вычисляет ID существующего Ed25519 ключа подписи из public key, поэтому `key-id.txt` не требуется в GitLab Variables.
- Восстановлен полный pipeline: validate, unit, smoke, полный E2E, security и расширенный DAST. Публикации GitHub mirror, GitHub Release и GHCR остаются ручными и доступны только после успешных quality gates.
- E2E-поднятие Compose-контура повторяет сетевую загрузку Docker-образов до трёх раз при кратковременном EOF Docker Hub. Повтор не применяется к уже начавшимся E2E-проверкам.
- Ручная публикация GitHub Release теперь доступна после успешных quality gates в pipeline ветки `main`: создаётся или проверяется тег версии, публикуется образ GHCR и добавляются release assets. В конце текста релиза выводится таблица результатов E2E и DAST.
- После успешных проверок доступны ручные jobs зеркалирования кода в GitHub и публикации подписанного GitHub Release с Docker-образом в GHCR.
- GitHub Release теперь содержит единый `release-evidence.md/json` и архив исходных очищенных E2E/DAST-доказательств. Публикация завершается ошибкой, если E2E содержат fail/skip или DAST сообщает High/Critical.
- Перед публикацией образа проверяются сгенерированные Nginx-артефакты, подпись, SBOM, provenance и `/healthz` собранного Docker-образа.
