## [1.2.6] - 25.06.2026

### Запросы
- Логирование запросов теперь маппит каждый настроенный хост и alias на его реальный `site_id` в сгенерированном `nginx base.conf`, поэтому трафик сервисов вроде `sentry.hantico.ru` больше не проваливается в `_global` и начинает корректно появляться во вкладке запросов, фильтре сервисов, anti-bot сценариях и per-site телеметрии защиты.

### UI
- Каталог `Ревизии` теперь отделяет историю конкретного сервиса от глобальных snapshot-ов: карточки и модалка теперь показывают только те ревизии, где менялся сам выбранный хост, а не любая последняя сборка runtime.
- Отключенные сервисы в списке больше не помечаются как `Ошибка`: вебморда теперь показывает корректный неактивный статус, а удаление easy-site сервиса дочищает связанный easy-profile через отдельный `DELETE /api/easy-site-profiles/:id`.
- Пресеты профиля сервиса больше не перетирают кастомный `limit_req_url` вроде `/auth/login/sentry/` обратно в `/` или `/api/` при повторном рендере формы профиля.

### Ядро
- `GET /api/requests` и `GET /api/events` теперь при кратковременной деградации runtime telemetry после успешного чтения возвращают последний валидный snapshot вместо `502/500`, что снижает ложные падения burst/smoke проверок и делает вебморду устойчивее под кратким всплеском нагрузки.

### Requests
- Для management-site custom easy-locations на `GET /api/requests` и `GET /api/events` больше не проксируют трафик в UI upstream: exact API-matches теперь остаются на `control-plane`, поэтому вкладка запросов снова открывается стабильно и проходит burst/smoke-проверки.
- Стартовая выборка вкладки `Запросы` уменьшена с `2000` до `200` записей за один запрос, чтобы вебморда больше не подвисала на многомегабайтных ответах `/api/requests` при открытии страницы.
- Вкладка `Запросы` теперь запрашивает данные с учётом локального часового пояса браузера и подтягивает имена сервисов также из easy-site профилей, поэтому запросы на границе суток и новые домены вроде `sentry.hantico.ru` больше не должны пропадать из таблицы и фильтра сервиса.
- `runtime /requests` теперь интерпретирует параметр `day` в локальном часовом поясе клиента через `tz_offset_minutes`, а не как жёсткий UTC-день, поэтому backend возвращает корректное окно запросов для вебморды.

### Логирование
- Healthcheck контейнеров больше не поднимает как отдельные проблемы два известных ложных сигнала: разовый startup `runtime security collector failed` на fallback к `security-events` и одиночные `OpenSearch JvmGcMonitorService overhead` предупреждения.
- Для OpenSearch-дневных индексов в вебморде логирования `size_bytes` больше не остаётся нулём: runtime теперь оценивает размер дневного среза пропорционально `store.size` общего индекса.

### UI
- Админская вебморда больше не держит сессию в фоне у скрытой вкладки: keep-alive ping и polling уведомлений останавливаются, пока вкладка не видна.

### Ядро
- TTL админской сессии control-plane сокращён до `60` минут простоя по умолчанию и теперь синхронизирован между серверной сессией, SSO-сессией и auth-cookie.
### UI
- Сохранение easy-site профиля теперь выполняется в одном batched-проходе без промежуточного auto-apply: базовые ресурсы, access policy и easy-profile сохраняются с `X-WAF-Auto-Apply-Disabled`, после чего UI запускает ровно один явный `compile/apply`, чтобы одна кнопка «Сохранить» больше не порождала две ревизии подряд.

### Диагностика
- `scripts/collect-waf-errors.sh` теперь дополнительно собирает `revision_apply_focus.txt` и счетчики compile/apply, записей easy-profile и операций с access policy, чтобы по одному архиву было проще увидеть цепочку «сохранение сервиса -> ревизия -> применение».
