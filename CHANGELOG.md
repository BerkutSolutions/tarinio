## [1.2.3] - 22.06.2026

### API
- Снижен шум вокруг `GET /api/events`: для UI-прокси отключено буферирование ответа, чтобы NGINX не засорял логи временными файлами при чтении длинной ленты событий.
- Убрано ложное runtime-предупреждение про `waf_site_id` в access log для запросов, которые не попадают в site-specific server block.

### Ядро
- Исправлена загрузка страницы `/services`: восстановлен корректный экспорт `ensureControlPlaneAccessManagementMethods`, а `normalizeAPIPositiveEndpointPolicies` возвращён к совместимой сигнатуре без внешнего callback.
- Для страницы сервисов добавлено расширенное логгирование клиентских ошибок: теперь в диагностику попадают `window.error`, `window.unhandledrejection` и подробный alert с этапом сбоя (`load`/`render`).
- В runtime-шаблонах NGINX скорректирован `Server-Timing`: fallback-страницы `4xx/5xx` теперь показывают реальные значения `rid/ip/ts` из `$request_id`, `$remote_addr` и `$msec`.
- Исправлен жизненный цикл auth-сессий при рестарте control-plane: boot-токен больше не фиксируется на `pepper`, поэтому активные браузерные сессии после перезапуска корректно инвалидируются.
- Для site-runtime добавлен отдельный `location ^~ /static/` без site-rate-limit include, чтобы загрузка UI-модулей не срывалась ложными `429`.
- Для локального `default`-профиля повышен `WAF_DEFAULT_LIMIT_REQ_RATE` до `120r/s`, чтобы штатная загрузка SPA и healthcheck не упирались в ложные лимиты на `localhost`.
- Для динамических импортов в `app.js` включён cache-busting (`?v=...`), чтобы браузер не держал старые ESM-модули после hotfix и перезапуска контейнеров.
- В release-скрипте улучшена публикация релиза: добавлена публикация GitHub Release по заметкам из `CHANGELOG.md` и отдельный режим `publish` для связки `release + docker`.
- Docker-публикация в release-скрипте теперь корректно ориентируется на версию `1.2.3` из `control-plane/internal/appmeta/meta.go`.

### Сервисы
- Страница `/services` восстановлена до рабочего состояния: устранены ошибки загрузки, падения динамических импортов и проблемы рендера detail-экрана.
- Исправлен `sites.detail-render-view.js`: удалён дублированный блок с повторным объявлением `SETTINGS_SEARCH_INDEX` и восстановлен корректный `return` шаблона.
- Исправлен `sites.detail-render-view-part2.js`: устранён синтаксический обрыв шаблона, который вызывал `Unexpected end of input`.
- Устранена финальная причина падения вкладки сервисов: в detail-view был обрезан хвост шаблонной строки после `renderDetailViewRuntimeTail(...)`.
- Для стабилизации работы `/services` в Firefox добавлен compatibility-fallback detail-рендера и облегчён preflight сервисных модулей.
- Таблица сервисов на `/services` снова работает стабильно: загрузка переведена на прямое чтение `/api/sites` без зависимости от проблемной цепочки `sites.page-main-*`.
- Для сохранения полного функционала (`list`, `edit`, `actions`) страница сервисов временно использует проверенный основной `sites.js`, а новая модульная цепочка сохранена в кодовой базе для отдельного этапа доводки.
- Выполнена модульная декомпозиция логики страницы сервисов: routing, normalize, auth/geo, import/export, access/resource upsert, list-view и detail-логика вынесены в отдельные файлы с более понятными именами.
- Переименованы неудобные имена модулей вида `sites.split-*` в нормальные рабочие имена вроде `sites.routing-merge.js`, `sites.normalize.js`, `sites.detail-draft.js`, `sites.detail-certs-bulk.js`.
- Исправлены ошибки декомпозиции в админке: возвращены корректные экспорты `renderUsersTable` и `renderRolesTable`, а также устранён сбой импорта `renderRolesTable` в `administration.js`.

### Профили безопасности
- Реализована строгая семантика `security_mode` для easy-профилей сервисов:
  - `transparent` — полностью пассивный режим без enforcement,
  - `monitor` — только наблюдение и логи,
  - `block` — полное применение включённых защитных модулей.
- Политика `security_mode` теперь влияет не только на основной easy-config, но и на генерацию custom rate-limit артефактов, чтобы в `transparent` и `monitor` не создавались блокирующие правила `limit_req`.
- Применение `security_mode` перенесено в серверную нормализацию easy-профиля: сохранённый профиль и legacy-policy sync теперь всегда соответствуют реальной семантике режимов, а не только стадии компиляции.

### UI
- Обновлён визуал карточек `Администрирование -> Скрипты`: улучшены иерархия, отступы, состояния и действия запуска/скачивания.
- Пересобран layout блока `Редактирование сервиса -> Общие HTTP`: списки методов и SSL-протоколов получили более стабильную сетку, ограничение высоты и более читаемое размещение чекбоксов `HTTP2/HTTP3`.
- Для точечной стилизации list-редакторов добавлены семантические hook-классы без изменения бизнес-логики.
- Переименованы подписи режимов безопасности в редакторе сервиса:
  - `Под наблюдением` -> `Наблюдение`
  - `Заблокировать` -> `Блокировка`
- Эти подписи синхронизированы во всех локалях `ru/en/de/sr/zh`.

### Тесты и стабильность
- Исправлена хрупкость archive-тестов launcher: тестовые кейсы fallback и фильтрации management-трафика переведены на относительные временные метки внутри retention-окна, поэтому общий `go test ./...` больше не начинает падать по мере устаревания фиксированной даты.
- Расширен e2e-smoke регресс `TestE2EUIFullRegression`: добавлены проверки `/services`, `/revisions`, `/profile` и жёсткая проверка загрузки page-модулей без `429` и HTML fallback.
- Добавлены и обновлены тесты компилятора для проверки реального поведения режимов `transparent`, `monitor` и `block`.
- Добавлен e2e-тест `TestE2ESecurityModesReality`, который прогоняет все 3 режима через API easy-профиля и проверяет фактические флаги защиты и состояние legacy-политик.
- Этот e2e-чек включён в обязательные `Startup Test Checks` release-скрипта.
