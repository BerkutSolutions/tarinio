# Evidence And Releases

This page belongs to the current documentation branch.

This document describes the evidence-grade controls implemented in TARINIO `2.0.10`: audit chaining, revision signatures, support bundles, and release artifacts.

## What Is Implemented

TARINIO `2.0.10` provides:

- tamper-evident audit chaining
- revision manifest signatures
- signed support bundle export
- generated release artifacts with checksums, `SBOM`, provenance, and signature

## Tamper-Evident Audit Chain

Each audit event stores:

- `prev_hash`
- `hash`

The hash is computed over the normalized audit payload plus the previous event hash. This creates a chained record set where a modification in the middle of the sequence breaks verification.

The chain is exposed in exported evidence rather than hidden behind an internal-only representation.

## Revision Signatures

When a revision is compiled, TARINIO signs the revision manifest fields that identify:

- revision ID
- version
- creation time
- checksum
- compiler identity
- approval policy state

The revision stores:

- `signature`
- `signature_key_id`

These values are returned by the revision catalog and included in support evidence.

## Approval And Rollout Correlation

The enterprise flow ties together:

- change creation
- revision compile
- approval
- apply
- rollback
- outcome

This correlation is visible through:

- `GET /api/revisions`
- `GET /api/audit`
- `GET /api/events`

The revision catalog shows:

- compiler identity
- approval state and approvals
- last apply result
- last apply time
- last rollout-related event

## Signed Support Bundle

Support evidence is exported through:

- `GET /api/administration/support-bundle`

The archive contains:

- `manifest.json`
- `signature.json`
- `evidence/public_key.pem`
- `audits.json`
- `events.json`
- `jobs.json`
- `revisions.json`

### Manifest Contents

The support bundle manifest includes:

- generation timestamp
- evidence key ID
- file inventory
- per-file `sha256` digests
- audit chain summary
- revision signature counts

`signature.json` signs `manifest.json` with the evidence `Ed25519` key stored by the enterprise settings store.

### Verification Workflow

To verify a support bundle:

1. Extract the archive.
2. Verify the `sha256` of each bundled file against `manifest.json`.
3. Verify the `Ed25519` signature in `signature.json` with `evidence/public_key.pem`.
4. Recompute the audit chain from `audits.json`.
5. Verify revision signatures from `revisions.json` if a deeper rollout review is required.

## Release Artifacts

Release evidence is generated by:

- `scripts/generate-release-artifacts.ps1`
- `scripts/release.ps1`
- `go run ./cmd/release-artifacts`

Artifacts are written to:

- `build/release/<version>/`

The generated set includes:

- `release-manifest.json`
- `signature.json`
- `checksums.txt`
- `sbom.cdx.json`
- `provenance.json`
- `release-public-key.pem`

### Release Manifest

`release-manifest.json` records:

- product version
- git tag
- commit SHA
- docker tags
- source input digests
- generated file digests
- signing key ID

### SBOM

`sbom.cdx.json` is a generated `CycloneDX 1.5` bill of materials assembled from:

- `go.mod`
- `package.json`
- `package-lock.json`

It is intended to give operators and reviewers a machine-readable dependency inventory for the exact release.

### Provenance

`provenance.json` records:

- release version
- tag and commit
- builder identity
- source inputs
- generated artifact context

It is a lightweight release attestation for promotion pipelines and audit review.

## Signing Keys

Release artifact signing uses a persistent local `Ed25519` key stored under:

- `.work/release-signing/`

Support-bundle signing uses the enterprise evidence key stored in the enterprise settings state.

These are intentionally separate concerns:

- product evidence key: operator-side support and audit evidence
- release signing key: build and promotion evidence

## Operational Guidance

For preprod and prod promotions:

1. Generate release artifacts.
2. Archive `release-manifest.json`, `signature.json`, `sbom.cdx.json`, and `provenance.json` together with the build ticket.
3. Promote only artifacts whose checksums and signature have been verified.
4. Export a signed support bundle after rollout if the change is high risk.

## CI/CD Quality Gates

In the `2.0.10` release line, the minimum release pipeline includes:

- `go test ./...` and `go vet ./...`;
- wiki/i18n quality tests in `ui/tests`;
- docs-site build verification;
- release artifact generation and file-contract checks;
- security checks (`govulncheck`, `npm audit`, `trivy`, secret scan).
