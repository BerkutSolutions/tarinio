# ADR-001: Runtime / Control-Plane Split

Status: Accepted
Date: 2026-03-31

## Context

The standalone WAF MVP consists of:
- a runtime data plane based on NGINX + ModSecurity + OWASP CRS
- a control plane that stores product state and manages configuration lifecycle
- a UI that operates only through control-plane APIs

Without a strict split, the product would drift into a mixed system where:
- runtime files become the real configuration source
- UI and API invent behavior before compiler and deployment contracts are fixed
- rollout, validation, and rollback become hard to reason about
- later SCC integration risks coupling directly to runtime internals

## Decision

The product is split into two mandatory backend concerns:
- `runtime`: traffic processing and enforcement only
- `control plane`: source-of-truth state, policy management, compilation, and rollout orchestration

The following rules are mandatory:
- responsibility mixing between runtime and control plane is prohibited
- runtime must never be treated as the source of truth
- runtime configuration must exist only as compiled output derived from control-plane state
- runtime behavior must depend on compiler output, not on operator-edited live config
- UI and API must target the control-plane model, not runtime internals

## Why They Are Separated

The split exists because the runtime and the control plane solve different problems:
- runtime must be optimized for stable request handling, enforcement, and safe reload
- control plane must be optimized for persistence, validation, auditability, and controlled change management

Keeping them separate solves these concrete problems:
- prevents drift between operator intent and active runtime files
- makes config generation deterministic and reviewable
- allows validation before activation and rollback after failed activation
- keeps request-path code free from admin and storage concerns
- allows the API and UI to remain stable even if runtime templates evolve
- preserves a clean standalone architecture without SCC dependency in MVP

## Problems This Decision Prevents

This decision explicitly prevents:
- direct editing of runtime config as the main product workflow
- storing canonical product state inside NGINX, ModSecurity, or mounted runtime files
- API handlers writing ad hoc runtime files without compiler/revision control
- UI screens defining backend behavior before the backend contract exists
- runtime-side business logic becoming the hidden control plane

## Constraints Imposed by This Decision

This split imposes the following architectural constraints:
- all operator-editable config must be stored in control-plane data structures
- all runtime config must be generated by the compiler from control-plane state
- apply, reload, health-check, and rollback must be orchestrated by the control plane
- runtime may expose status and logs, but may not own revision history or policy state
- UI may not call runtime directly for configuration changes
- API may not bypass the compiler by writing live runtime config as a primary path

## Consequences for Stage 1

The mandatory Stage 1 implementation order is:
1. config compiler
2. config validation and apply pipeline
3. runtime wiring with NGINX and ModSecurity
4. control-plane API
5. UI

This ADR imposes the following Stage 1 restrictions:
- runtime work cannot proceed as meaningful product implementation before compiler contracts exist
- API work cannot invent configuration behavior outside the compiler-backed model
- UI work cannot define data contracts or runtime semantics by screen design
- site, policy, TLS, access-control, and rate-limit features must end in compiled runtime output
- rollout status, revision state, and audit flow must come from control-plane ownership

## Rejected Alternative

Rejected alternative: use runtime files and runtime state as the operational source of truth, then build API/UI around that state.

Why rejected:
- it makes validation and rollback less reliable
- it creates hidden state outside the product model
- it weakens auditability and repeatability
- it encourages direct file mutation instead of controlled compilation
- it increases coupling between admin workflows and runtime internals

## Resulting Rule

If a feature requires runtime behavior, it must first be expressible as:
- control-plane state
- compiler transformation
- revisioned runtime output
- controlled apply flow

If it cannot satisfy that chain, it is not ready for Stage 1 implementation.
