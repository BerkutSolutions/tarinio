# Высокая доступность и многонодовый режим

Эта страница относится к текущей ветке документации.

В версии `2.0.3` TARINIO поддерживает практический многонодовый режим для control-plane. Это позволяет запускать несколько узлов за балансировщиком API и сохранять единое состояние системы.

## Что входит в HA-схему

- общее состояние в PostgreSQL;
- распределённая координация через Redis;
- несколько узлов `control-plane` за API-балансировщиком;
- выбор лидера для фоновых задач;
- глобальная сериализация операций `compile` и `apply`.

Это означает, что прежнее ограничение на работу только в одном узле больше не соответствует актуальной архитектуре control-plane.

## Поддерживаемая топология

```text
ui -> api-lb -> control-plane-a / control-plane-b
                    |             |
                    +---- Redis ---+
                    +-- PostgreSQL-+
                    +-- shared runtime root
                              |
                           runtime
                              |
                        защищаемые сайты
```

В этой схеме:

- любой узел control-plane может обслуживать API-запросы;
- фоновые задачи, требующие единственного исполнителя, выполняет только лидер;
- операции `compile` и `apply` защищены распределёнными блокировками;
- все узлы работают с одним и тем же состоянием в PostgreSQL.

## Что реализовано

### PostgreSQL как основное хранилище

Состояние control-plane хранится в PostgreSQL. Если система ранее использовала файловое состояние, оно переносится автоматически при первом запуске.

### Redis для координации

Через Redis реализованы:

- блокировка на время компиляции ревизии;
- блокировка на время применения ревизии;
- выбор лидера для `auto-apply`;
- выбор лидера для `dev fast start`;
- выбор лидера для `tls auto-renew`.

### Идентификация активного узла

Маршрут `GET /api/app/meta` возвращает:

- `ha_enabled`;
- `ha_node_id`.

Это позволяет проверить, какой узел отвечает через балансировщик в конкретный момент времени.

## Обязательные условия для HA

Все узлы control-plane должны использовать:

- одинаковый `POSTGRES_DSN`;
- одинаковый `CONTROL_PLANE_REDIS_ADDR`;
- общий токен доступа к runtime API;
- общий путь к артефактам runtime или эквивалентное общее хранилище.

При этом у каждого узла должен быть собственный:

- `CONTROL_PLANE_HA_NODE_ID`.

Ключевые переменные режима HA:

- `CONTROL_PLANE_HA_ENABLED=true`
- `CONTROL_PLANE_HA_OPERATION_LOCK_TTL_SECONDS`
- `CONTROL_PLANE_HA_LEADER_LOCK_TTL_SECONDS`

## Поведение кластера

### Компиляция и применение ревизий

В каждый момент времени только один узел может удерживать глобальную блокировку операции. Это предотвращает:

- гонки между версиями ревизий;
- параллельную запись в путь публикации;
- расхождение состояния при применении.

### Автоприменение

Изменения могут прийти на любой узел, но фактическое автоприменение выполняет только текущий лидер.

### Автозапуск в средах разработки

Если включён `dev fast start`, операция также выполняется только лидирующим узлом.

### Автопродление сертификатов

Продление сертификатов выполняется через механизм выбора лидера, чтобы исключить дублирование заданий.

## Готовая лаборатория для проверки HA

Для локальной проверки используйте профиль:

- `deploy/compose/ha-lab`

В него входят:

- `api-lb`
- `control-plane-a`
- `control-plane-b`
- `postgres`
- `redis`
- `ui`
- `runtime`
- `ddos-model`
- `demo-app`
- `toolbox`

Профиль рассчитан на локальную рабочую станцию и позволяет проверить отказоустойчивость без отдельной инфраструктуры.

## Базовый сценарий проверки

1. Запустите лабораторию:

```powershell
cd deploy/compose/ha-lab
docker compose --profile tools up -d --build
```

2. Откройте `http://localhost:18080/login`.
3. Разверните 20 демонстрационных сервисов:

```powershell
docker compose --profile tools exec toolbox /tools/provision-20-services.sh
```

4. Проверьте распределение запросов по узлам:

```powershell
docker compose --profile tools exec toolbox /tools/cluster-status.sh
```

5. Запустите небольшую нагрузку:

```powershell
docker compose --profile tools exec toolbox /tools/mini-ddos.sh
```

6. Остановите один узел `control-plane` и убедитесь, что API продолжает работать через оставшийся.

## Текущие границы реализации

Что уже поддерживается:

- высокая доступность control-plane;
- единое состояние системы;
- координация между несколькими узлами;
- локальная лаборатория для проверки отказоустойчивости и Anti-DDoS.

Что пока не входит в релиз:

- многорегионное управление;
- централизованное управление флотом инсталляций;
- полноценная абстракция распределённого хранилища без внешнего общего слоя;
- интеграции уровня `SIEM`, `SSO`, `SCIM`.
