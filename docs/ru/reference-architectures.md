# Референсные архитектуры

Эта страница относится к текущей ветке документации.

## Назначение

Эта страница даёт типовые схемы развёртывания, чтобы оператору не приходилось собирать топологию с нуля.

## 1. Стартовый однокомпонентный контур

Использовать, когда:

- идёт знакомство с TARINIO;
- защищается небольшое число сервисов;
- высокая доступность пока не требуется.

Состав:

- один control-plane;
- один рантайм;
- встроенный PostgreSQL;
- при желании локальная наблюдаемость.

## 2. Control-plane в режиме высокой доступности

Использовать, когда:

- важна доступность административного контура;
- нужны поэтапные обновления без простоя API;
- нескольким операторам нужен стабильный доступ к control-plane.

Состав:

- `api-lb`
- `control-plane-a`
- `control-plane-b`
- общий PostgreSQL
- общий Redis
- рантайм

## 3. Сегментированное развёртывание корпоративного уровня

Использовать, когда:

- административный доступ должен быть отделён;
- сервисы данных должны оставаться внутренними;
- наблюдаемость выносится в отдельный контур мониторинга.

## 4. Лабораторная архитектура для проверки

Использовать, когда:

- проверяется отказоустойчивость;
- снимаются результаты бенчмарков;
- тестируются сценарии обновления и миграции.

Состав:

- `deploy/compose/ha-lab`
