# Архитектура TARINIO 2.0.0

Wiki baseline: `2.0.0`

TARINIO — self-hosted платформа управления и защиты веб-трафика, где control-plane описывает желаемое состояние, а runtime исполняет только собранную и применённую конфигурацию.

## Архитектурная модель

Продукт состоит из четырёх практических слоёв:

- `UI`: операторский интерфейс, onboarding, login, healthcheck.
- `Control-plane`: API, RBAC, хранилище конфигурации, аудит, каталог ревизий, orchestration изменений.
- `Compiler / revision pipeline`: сборка bundle из бизнес-сущностей и профилей.
- `Runtime`: NGINX + ModSecurity + OWASP CRS + прикладные защитные механизмы.

## Базовые принципы

- Source of truth для намерений оператора находится в control-plane.
- Runtime не редактируется вручную и не считается источником истины.
- Любое изменение конфигурации должно быть переводимо в ревизию.
- Rollout и rollback рассматриваются как штатные операции, а не как аварийные хаки.
- Контроль доступа и проверка прав выполняются сервером на каждом endpoint.

## Ключевые домены

В версии `2.0.0` архитектурно важны следующие домены:

- сайты и upstream;
- TLS и сертификатные материалы;
- access, WAF и rate-limit политики;
- easy site profiles как высокоуровневый прикладной слой конфигурации;
- Anti-DDoS;
- ревизии, snapshots и apply jobs;
- audit и observability данные;
- runtime settings и retention.

## Жизненный цикл изменения

Стандартный цикл изменения выглядит так:

1. Оператор меняет сущности через UI, API или CLI.
2. Control-plane сохраняет целевое состояние.
3. Создаётся revision через compile.
4. Revision проходит validate/apply в runtime-контуре.
5. Система хранит историю применения и поддерживает rollback.

Именно поэтому раздел `Ревизии` является ключевой частью продукта, а не просто историческим журналом.

## Почему ревизии важны

В `2.0.0` revision subsystem включает:

- агрегированный каталог ревизий по сервисам;
- active/pending/failed классификацию;
- timeline статусов и событий применения;
- удаление неактивных ревизий;
- сохранение последнего результата применения у ревизии даже после очистки timeline.

Это делает change-management воспроизводимым и пригодным для эксплуатации.

## Runtime boundary

Runtime отвечает за:

- проксирование и терминацию трафика;
- исполнение NGINX-конфигурации;
- ModSecurity/OWASP CRS enforcement;
- применение Anti-DDoS и сетевых ограничений;
- журналирование запросов и security events.

Control-plane отвечает за:

- хранение операторских намерений;
- авторизацию;
- сборку и публикацию конфигурации;
- аудит и API;
- UI.

## Интерфейс как архитектурный слой

UI в `2.0.0` не является тонкой формой над несколькими сущностями. Он отражает несколько уровней продукта:

- операционный обзор через `Dashboard`;
- конфигурационный слой через `Сайты`, `TLS`, `Anti-DDoS`, `OWASP CRS`;
- observability слой через `Запросы`, `События`, `Баны`, `Активность`;
- change-management слой через `Ревизии`;
- platform layer через `Настройки`, `Профиль`, `Администрирование`.
