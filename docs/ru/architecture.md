# Архитектура

Эта страница относится к текущей ветке документации.

TARINIO — это платформа для управления и защиты веб-трафика, разворачиваемая в собственной инфраструктуре. Control-plane хранит и проверяет желаемое состояние, а runtime применяет только уже собранную и опубликованную конфигурацию.

## Архитектурная модель

Платформа состоит из четырёх практических слоёв:

- `UI` — операторский интерфейс, первичная настройка, вход и рабочие экраны.
- `Control-plane` — API, RBAC, хранилище конфигурации, аудит и каталог ревизий.
- `Compiler / revision pipeline` — преобразование операторских настроек в готовые артефакты ревизии.
- `Runtime` — выполнение опубликованной конфигурации, обработка трафика и применение защитных механизмов.

## Базовые принципы

- Источником истины является control-plane, а не файловая система runtime.
- Любое значимое изменение должно быть переводимо в ревизию.
- Применение и откат считаются штатными операциями.
- Контроль доступа и проверка прав всегда выполняются на сервере.
- Ручное редактирование runtime не входит в поддерживаемую модель эксплуатации.

## Ключевые домены

В версии `2.0.4` архитектурно важны следующие области:

- сайты и upstream-сервисы;
- TLS и сертификатные материалы;
- политики доступа, WAF и ограничения скорости;
- упрощённые профили сайтов как прикладной уровень конфигурации;
- Anti-DDoS;
- ревизии, снимки состояния и задания применения;
- аудит, журналы и данные наблюдаемости;
- системные настройки и сроки хранения данных.

## Жизненный цикл изменения

Стандартный цикл изменения выглядит так:

1. Оператор изменяет настройки через интерфейс, API или CLI.
2. Control-plane сохраняет целевое состояние.
3. Компилятор формирует новую ревизию.
4. Ревизия проходит проверку и публикуется в runtime.
5. Система сохраняет историю применения и позволяет выполнить откат.

Именно поэтому раздел `Ревизии` — это рабочий инструмент управления изменениями, а не просто журнал.

## Почему ревизии важны

В `2.0.4` подсистема ревизий включает:

- единый каталог ревизий по сервисам;
- разделение на `active`, `pending` и `failed`;
- историю статусов и событий применения;
- удаление неактивных ревизий;
- сохранение последнего результата применения даже после очистки ленты статусов.

Это делает изменения воспроизводимыми и пригодными для эксплуатации в продуктивной среде.

## Граница между control-plane и runtime

Runtime отвечает за:

- приём и обработку трафика;
- применение сетевых и HTTP-правил;
- использование NGINX, ModSecurity и OWASP CRS;
- ведение журналов запросов и событий безопасности.

Control-plane отвечает за:

- хранение операторских настроек;
- авторизацию и модель ролей;
- сборку и публикацию артефактов;
- аудит;
- HTTP API и интерфейс администратора.

## Интерфейс как часть архитектуры

Интерфейс в TARINIO отражает несколько уровней платформы:

- операционный обзор через `Dashboard`;
- настройку конфигурации через `Сайты`, `TLS`, `Anti-DDoS`, `OWASP CRS`;
- анализ состояния через `Запросы`, `События`, `Баны`, `Активность`;
- управление изменениями через `Ревизии`;
- управление платформой через `Настройки`, `Профиль`, `Администрирование`.
