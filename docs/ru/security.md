# Безопасность TARINIO 2.0.1

Wiki baseline: `2.0.1`

Документ фиксирует минимальный production baseline и ключевые практики безопасной эксплуатации TARINIO.

## Security baseline

- `APP_ENV=prod`
- дефолтные секреты недопустимы;
- доступ к control-plane должен быть ограничен доверенными сетями;
- административный доступ должен идти по HTTPS;
- trusted proxies должны быть настроены явно;
- аудит и хранилище секретов должны быть защищены на уровне платформы.

## Аутентификация и доступ

В версии `2.0.1` платформа поддерживает:

- сессионный login;
- `2FA` через TOTP;
- passkeys для login и как второй фактор;
- серверную проверку permissions на каждом endpoint.

Рекомендации:

- использовать только именованные учётные записи;
- включать второй фактор для административных пользователей;
- удалять неиспользуемые passkeys;
- регулярно проверять, какие роли и permissions есть у операторов.

## Секреты и конфиденциальные данные

- не хранить `.env` в публичных репозиториях;
- держать секреты в отдельном защищённом хранилище;
- ограничить доступ к ключам и сертификатным материалам;
- не публиковать значения EAB, DNS API tokens и private keys в тикетах и чатах.

## Сетевая модель

- публиковать только необходимые порты;
- отделять ingress runtime от административного доступа;
- ограничивать доступ к PostgreSQL и внутренним сетям;
- использовать host firewall вместе с runtime L4 guard.

## TLS и сертификаты

Рекомендуется:

- использовать HTTPS для административного контура;
- контролировать сроки действия сертификатов;
- заранее проверять сценарий продления ACME;
- хранить экспортируемые сертификатные архивы как чувствительные артефакты;
- документировать, какой режим используется: import, self-signed, ACME `http-01` или `dns-01`.

## Безопасность изменений

- не редактировать runtime вручную;
- все рискованные изменения проводить через compile/apply;
- перед изменением защиты иметь готовую known-good ревизию;
- после изменения сразу проверять `Dashboard`, `События`, `Запросы` и `Баны`.

## Аудит и расследование

Для расследования используйте:

- `Активность` для административных действий;
- `События` для security detections;
- `Запросы` для request-level деталей;
- `Ревизии` для сопоставления инцидента с rollout.

## Backup и восстановление как часть безопасности

Безопасная эксплуатация предполагает:

- регулярный backup;
- минимум одну офф-хост копию;
- проверенный restore drill;
- понятный порядок отката после неудачного изменения.
