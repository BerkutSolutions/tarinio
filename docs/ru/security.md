# Безопасность (RU)

Базовая версия документации: `1.1.5`

## Production baseline

- `APP_ENV=prod`
- дефолтные секреты запрещены
- HTTPS обязателен (built-in TLS или trusted reverse-proxy TLS)
- ограничить `BERKUT_SECURITY_TRUSTED_PROXIES`
- включить tamper-evident аудит через `BERKUT_AUDIT_SIGNING_KEY`

## Работа с секретами

- Хранить `.env` вне публичных репозиториев.
- Ротировать админские и интеграционные секреты по графику.
- Не писать секреты в тикеты, чаты и логи.
- Ограничить доступ к файлам секретов на уровне хоста.

## Сетевая база

- Публиковать только необходимые порты.
- Ограничить доступ к control-plane доверенными сетями.
- Сохранять изоляцию docker network.
- Использовать host firewall и runtime L4 guard вместе.

## TLS и сертификаты

- Принудительный HTTPS для операторского и пользовательского трафика.
- Мониторинг сроков действия сертификатов.
- Регулярная проверка сценария продления.
- Ошибка выпуска сертификата рассматривается как incident.

## Контроль доступа

- Использовать только именованные операторские аккаунты.
- Оперативно удалять устаревшие admin-аккаунты.
- Регулярно проверять audit events для привилегированных действий.
- Придерживаться least privilege для host и CI/CD пользователей.

## Безопасность изменений

- Все security-sensitive изменения проводить через compile/apply revision.
- Не редактировать runtime filesystem вручную.
- Перед enforcement-изменениями держать готовый rollback путь.

## Готовность к инцидентам

- Держать проверенный backup/restore процесс.
- Держать known-good revision для аварийного отката.
- Вести таймлайн инцидента и remediation действия.
