# Безопасность

Эта страница относится к текущей ветке документации.

Документ описывает минимальный базовый уровень безопасности и практики, которые следует применять при эксплуатации TARINIO в рабочей среде.

## Базовые требования

- `APP_ENV=prod`
- не используйте секреты по умолчанию;
- ограничивайте доступ к control-plane доверенными сетями;
- публикуйте административный интерфейс только по HTTPS;
- явно настраивайте trusted proxies;
- защищайте аудит, секреты и сертификатные материалы на уровне платформы.

## Аутентификация и доступ

В версии `2.0.3` поддерживаются:

- вход по имени пользователя и паролю;
- `2FA` на основе TOTP;
- passkeys как основной или дополнительный фактор;
- серверная проверка прав на каждом маршруте API.

Рекомендуется:

- использовать персональные учётные записи вместо общих;
- включать второй фактор для административных пользователей;
- удалять неиспользуемые passkeys;
- регулярно проверять набор ролей и прав операторов.

## Работа с секретами

- не храните `.env` в публичных репозиториях;
- используйте отдельное защищённое хранилище для секретов;
- ограничивайте доступ к приватным ключам и сертификатам;
- не передавайте EAB, DNS API tokens и приватные ключи в тикетах и чатах.

## Сетевая модель

- публикуйте только необходимые порты;
- разделяйте контур входящего трафика и административный контур;
- ограничивайте доступ к PostgreSQL и внутренним сетям;
- используйте host firewall вместе с защитой runtime на сетевом уровне.

## TLS и сертификаты

Рекомендуется:

- использовать HTTPS для административного доступа;
- контролировать срок действия сертификатов;
- заранее проверять сценарий продления ACME;
- обращаться с экспортированными архивами сертификатов как с чувствительными артефактами;
- документировать выбранный режим выпуска и продления сертификатов.

## Безопасность изменений

- не редактируйте runtime вручную;
- проводите рискованные изменения только через ревизии;
- до изменения защитных политик держите под рукой заведомо рабочую ревизию;
- после применения изменений сразу проверяйте `Dashboard`, `События`, `Запросы` и `Баны`.

## Аудит и расследование

Для расследования используйте:

- `Активность` для действий операторов;
- `События` для срабатываний защит;
- `Запросы` для детального разбора трафика;
- `Ревизии` для связи инцидента с конкретным изменением.

## Резервное копирование как часть безопасности

Безопасная эксплуатация включает:

- регулярное резервное копирование;
- как минимум одну копию вне основного хоста;
- проверенный сценарий восстановления;
- понятный и отработанный порядок отката.
