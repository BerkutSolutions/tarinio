# Руководство по аварийному восстановлению

Эта страница относится к текущей ветке документации.

## Область

Этот документ описывает восстановление при потере ноды, сервисов данных и при полном восстановлении из backup.

## Приоритеты восстановления

Восстанавливать нужно в таком порядке:

1. persistent state;
2. доступность control-plane;
3. readiness runtime;
4. ingress validation;
5. observability и вторичные сервисы.

## Потеря одной control-plane ноды

Если есть HA:

- продолжить работу через оставшуюся ноду;
- восстановить упавшую ноду из deployment artifacts;
- вернуть её в кластер с теми же PostgreSQL и Redis.

Если single-node:

- восстановить хост;
- вернуть volumes;
- проверить `/healthz`, `/login` и `/healthcheck`.

## Потеря PostgreSQL

Основные действия:

- остановить небезопасную write-активность;
- восстановить PostgreSQL из последнего валидного backup;
- проверить schema и migrated control-plane state;
- подтвердить known-good revision после восстановления.

## Потеря Redis в HA

Основные действия:

- восстановить Redis;
- убедиться, что lock и leader behavior снова работают корректно;
- проверить отсутствие split-brain compile/apply активности.

## Полная пересборка хоста

Рекомендуемый порядок:

1. вернуть deployment artifacts;
2. восстановить PostgreSQL;
3. восстановить Redis, если он используется;
4. восстановить runtime и certificate volumes при необходимости;
5. запустить сервисы;
6. выполнить smoke validation.
