# Контракт миграции и совместимости

Этот документ фиксирует правила совместимости `2.0.8` для логгирования, миграции секретов и перехода между standalone и enterprise-контурами.

## Обратная совместимость

Существующие standalone-установки продолжают поддерживаться:

- старые runtime settings нормализуются в текущую модель;
- отсутствие опциональных backend’ов не ломает старт;
- legacy request archive остаётся доступным до завершения миграции;
- очистка выполняется только после подтверждения целевого backend’а.

Главное правило простое: обновление не должно оставлять систему в полу-рабочем состоянии.

## Правила миграции request archive

Для request-данных порядок такой:

1. локальный legacy-архив сохраняется;
2. legacy day-файлы импортируются в активный backend;
3. runtime проверяет, что ожидаемые записи реально присутствуют в backend;
4. только после этого удаляется подтверждённый legacy-файл;
5. состояние миграции сохраняется отдельно.

Это относится к:

- обновлению `2.0.5 -> 2.0.8`, где история запросов ещё жила в локальных `*.jsonl`;
- enterprise-сценариям, где подтверждённые данные дополнительно перемещаются из hot в cold tier.

## Правила cutover для backend’ов

Фактическая request-модель в `2.0.8` такая:

1. `OpenSearch` — primary backend для запросов, если он включён;
2. `ClickHouse` — optional cold tier для enterprise-истории;
3. локальный архив остаётся spool/fallback-слоем, а не основным источником чтения для UI;
4. если backend настроен неполно, продукт безопасно откатывается в fallback, а не делает вид, что backend активен.

## Правила миграции секретов

Для секретов логгирования порядок такой:

1. оператор включает `vault`;
2. TARINIO валидирует подключение к Vault и доступ по token;
3. backend-секреты записываются в Vault;
4. продукт проверяет, что секрет читается обратно;
5. legacy локальные копии очищаются.

Инвариант:

- сначала запись и проверка;
- потом очистка старой копии.

## Default и Enterprise

### Default

Штатный профиль `2.0.8 default` состоит из:

- `PostgreSQL`
- `OpenSearch`
- `Vault`
- без обязательного `Redis`
- без обязательного `ClickHouse`

### Enterprise

Enterprise-профиль добавляет:

- многонодовый control-plane
- `Redis` для HA-координации
- `ClickHouse` для cold analytics

Обновление должно оставаться безопасным в обоих режимах.

## Что считать успешной миграцией

Миграция считается успешной только если одновременно выполняется всё:

1. стек поднимается без ручной правки compose;
2. свежие запросы видны через API/UI;
3. legacy request-файлы мигрируются до удаления;
4. backend-секреты больше не остаются в небезопасных локальных копиях после миграции в Vault;
5. standalone сохраняет безопасный spool/fallback-путь на случай временной недоступности backend’а;
6. enterprise сохраняет validated hot-to-cold перенос истории.

## Что считать регрессией

Регрессией считаются:

- удаление legacy request archive до подтверждения данных в backend;
- состояние, где backend показан как активный, но runtime реально не может его использовать;
- зависимость UI от локальных request-файлов вместо API/backend-path;
- очистка локальных секретов до подтверждённой записи в Vault;
- ситуация, где standalone перестаёт жить без enterprise-only зависимостей.
