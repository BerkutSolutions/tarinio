# Контракт миграции и совместимости

Этот документ фиксирует правила совместимости для hot/cold-архитектуры логгирования и контура секретов, введённых в версии `2.0.5`.

## Обратная совместимость

Существующие standalone-установки продолжают поддерживаться:

- старые файловые архивы запросов не ломаются;
- ранее сохранённые runtime settings нормализуются в новую модель;
- отсутствие настроек `OpenSearch` или `Vault` не должно ломать старт системы;
- `ClickHouse` остаётся безопасным cold-слоем и при неполной конфигурации переводится в fallback.

Идея простая: обновление не должно превращать старую установку в полурабочий стенд.

## Правила cutover для backend-ов

Текущая модель переключения такая:

1. `file` остаётся локальным fallback-слоем;
2. `OpenSearch` используется как hot backend;
3. `ClickHouse` используется как cold backend;
4. если конфигурация backend-а неполная, продукт возвращается в безопасный fallback, а не остаётся в неопределённом состоянии.

## Правила миграции секретов

Для секретов логгирования порядок такой:

1. оператор выбирает `vault`;
2. TARINIO валидирует подключение к Vault и token;
3. backend-секреты записываются в Vault;
4. продукт убеждается, что секрет читается обратно;
5. локальные persisted-копии backend credentials очищаются.

Ключевой принцип:

- сначала успешная запись и проверка;
- потом удаление старой локальной копии.

## Безопасность обновления

Профиль `default` теперь включает `OpenSearch`, `ClickHouse` и `Vault` как штатную модель релиза.

При этом обновление должно оставаться безопасным:

- состояние `OpenSearch` не разрушает уже существующие файловые архивы запросов;
- локальный request archive продолжает работать как слой совместимости;
- `Vault` в default profile теперь bootstrap-ится автоматически, а не остаётся “контейнером без готовности”.

## Что считать успешной миграцией

Миграцию можно считать успешной, если выполняются все условия:

1. стек поднимается без ручного исправления compose;
2. `OpenSearch`, `ClickHouse` и `Vault` healthy;
3. свежие запросы видны в UI;
4. исторические запросы доступны через cold storage;
5. секреты backend-ов больше не висят в локальной открытой копии;
6. fallback-архив всё ещё доступен как совместимый резервный слой.

## Что считать регрессией

Регрессией следует считать:

- запуск, который требует ручного init/unseal Vault на штатном default profile;
- потерю доступа к старым request archives после обновления;
- ситуацию, когда UI показывает новый backend, но runtime реально не может его использовать;
- очистку локальных секретов до подтверждённой записи в Vault;
- состояние “вроде включено, но данные никуда не пишутся”.

## Практический вывод

Контракт `2.0.5` такой:

- новая архитектура должна включаться безопасно;
- старые установки не должны ломаться;
- логгирование должно деградировать предсказуемо;
- контур секретов должен мигрировать без потери доступа и без сохранения старых небезопасных копий.
