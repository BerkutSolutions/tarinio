# Архитектура и принципы Anti-DDoS модели

Этот документ описывает, как устроен контур Anti-DDoS в TARINIO, из каких слоёв он состоит, на какие сигналы опирается и как принимает решения.

## 1. Общая схема

Anti-DDoS в TARINIO — это не одна настройка, а три взаимосвязанных слоя:

1. `L4 guard`
2. глобальное ограничение HTTP-трафика на уровне L7
3. адаптивная модель

Роли этих слоёв различаются:

- `L4 guard` ограничивает вредный TCP-трафик ещё до обработки запроса приложением;
- L7-ограничение удерживает базовый поток HTTP-запросов в предсказуемых рамках;
- адаптивная модель реагирует на поведение источников и временно усиливает защиту.

## 2. Источник истины и путь до runtime

Исходные настройки хранятся в control-plane:

- `anti_ddos_settings.json`

Дальше изменения проходят обычный путь публикации:

1. оператор сохраняет настройки в control-plane;
2. выполняется компиляция ревизии;
3. создаётся снимок состояния;
4. выполняется проверка артефактов и контракта runtime;
5. ревизия применяется и становится активной.

В итоговый набор артефактов входят как минимум:

- `l4guard/config.json`
- `ddos-model/config.json`

Runtime не читает прикладную модель напрямую. Он использует только артефакты активной ревизии.

## 3. Что делает `L4 guard`

`L4 guard` запускается до старта NGINX и применяет детерминированные правила `iptables`.

Базовая логика включает:

- ограничение числа одновременных TCP-соединений на один IP через `connlimit`;
- ограничение частоты новых соединений через `hashlimit`;
- применение `DROP` или `REJECT` при превышении порога.

Это защищает от:

- всплеска параллельных соединений;
- слишком высокой частоты новых подключений;
- злоупотребления keep-alive, когда один источник удерживает чрезмерно много сокетов.

`L4 guard` работает в цепочке:

- `DOCKER-USER`, если доступен docker-aware сценарий;
- иначе `INPUT`.

В режиме `docker bridge` или `DNAT` используется `destination_ip`, чтобы правила попадали на правильный post-DNAT адрес контейнера runtime.

## 4. Что делает глобальное ограничение L7

Если включён `enforce_l7_rate_limit`, компилятор добавляет единое ограничение HTTP-запросов для всех включённых сайтов, кроме административного интерфейса.

Назначение этого уровня:

- не позволять одному источнику держать чрезмерное HTTP-давление;
- задавать базовый потолок нагрузки ещё до включения адаптивной эскалации.

Основные параметры:

- `l7_requests_per_second`
- `l7_burst`
- `l7_status_code`

Это статический уровень защиты в пределах активной ревизии. Он не изменяется сам по себе.

## 5. Откуда адаптивная модель получает данные

Адаптивная модель работает отдельным процессом `ddos-model`.

Она читает:

- журнал доступа runtime;
- текущее состояние модели из `model-state.json`;
- конфигурацию активной ревизии из `ddos-model/config.json`.

На выходе модель формирует:

- `adaptive.json` с временными IP-правилами.

Далее `L4 guard` читает `adaptive.json` и расширяет базовые правила временными действиями `throttle` или `drop`.

## 6. Какие сигналы учитывает модель

Модель реагирует не на любой всплеск, а на признаки вредоносного или избыточного поведения.

Базовые сигналы по HTTP-статусам:

- `429` — вес `model_weight_429`, обычно указывает на давление по скорости запросов;
- `403` — вес `model_weight_403`, чаще связан с запрещённым доступом или отклонённым шаблоном;
- `444` — вес `model_weight_444`, как правило указывает на явно нежелательный трафик или раннее принудительное закрытие соединения.

Дополнительные аварийные сигналы:

- общий `RPS` выше `model_emergency_rps`;
- число уникальных IP за секунду выше `model_emergency_unique_ips`;
- `RPS` одного IP выше `model_emergency_per_ip_rps`.

## 7. Как считается `score`

Для каждого IP модель хранит:

- текущий `score`;
- время последнего события;
- время последнего обновления;
- текущую стадию;
- `expires_at`.

Когда приходит новый сигнал:

- к IP добавляется соответствующий вес;
- перед этим применяется экспоненциальное затухание по `model_decay_lambda`;
- если источник долго ведёт себя спокойно, старый вклад постепенно уменьшается.

Это позволяет:

- не помнить единичный случайный всплеск слишком долго;
- естественно снижать риск для затихшего источника;
- усиливать реакцию только при повторяемом или интенсивном сигнале.

## 8. Пороги и стадии реакции

Модель использует две основные стадии:

- `throttle`, если `score >= model_throttle_threshold`
- `drop`, если `score >= model_drop_threshold`

Если `score` ниже порога `throttle`, IP не попадает в итоговый адаптивный вывод.

Если `score` достигает порога `throttle`, в `adaptive.json` появляется запись с действием `throttle`.

Если `score` достигает порога `drop`, появляется запись с действием `drop`.

Порог `drop` всегда должен быть выше порога `throttle`.

## 9. Аварийные детекторы

### 9.1 Всплеск распределённого шума

Модель одновременно анализирует:

- количество запросов за секунду;
- число уникальных IP в ту же секунду.

Если одновременно превышены оба порога, модель:

- повышает `score` всем IP из всплеска;
- в крайних случаях может сразу применить `drop`.

Это помогает при распределённой атаке, когда отдельный IP выглядит умеренно, но суммарная картина уже опасна.

### 9.2 Агрессивный одиночный источник

Для каждого IP считается `RPS` по секундам.

Если один адрес превышает аварийный порог, модель:

- быстро усиливает реакцию;
- при сильном превышении может сразу перейти к `drop`, минуя медленное накопление.

## 10. Почему итог агрегируется по IP

Внутри модель может учитывать контекст `site + ip`, но при публикации в `adaptive.json` итог агрегируется по IP.

IP попадает в итоговый набор, если:

- у него есть глобальный маркер;
- или вредоносный шаблон проявился хотя бы на двух сайтах.

Это нужно, чтобы:

- снизить риск ложной блокировки по одному локальному эпизоду;
- считать межсайтовый шум сильным признаком автоматизированной атаки.

## 11. Что содержит `adaptive.json`

Файл содержит:

- `updated_at`
- `throttle_rate_per_second`
- `throttle_burst`
- `throttle_target`
- `entries[]` с полями `action` и `expires_at`

Возможные действия:

- `throttle`
- `drop`

После этого `L4 guard` переводит эти записи в конкретные правила `iptables`.

## 12. Как `L4 guard` применяет адаптивные записи

Если запись имеет действие `drop`, создаётся прямое правило блокировки IP.

Если запись имеет действие `throttle`, создаётся правило `hashlimit` с параметрами:

- `model_throttle_rate_per_second`
- `model_throttle_burst`
- `model_throttle_target`

Сама модель не применяет firewall-правила напрямую. Она публикует решение, а сетевое применение делает `L4 guard`.

## 13. Время удержания

После эскалации IP получает `expires_at`, рассчитанный по `model_hold_seconds`.

Это означает:

- защита не исчезает мгновенно после короткого спада трафика;
- система избегает быстрых циклов включения и выключения мер.

После истечения удержания и снижения `score` запись удаляется из состояния и исчезает из `adaptive.json`.

## 14. Почему модель учитывает операторские пределы

Расчёт эффективных аварийных порогов учитывает уже заданные ограничения:

- `rate_per_second`
- `l7_requests_per_second`
- `conn_limit`

Это нужно, чтобы модель:

- не начинала эскалацию слишком рано;
- не противоречила настройкам, которые оператор задал сознательно;
- оставалась согласованной с общей политикой защиты.

## 15. Как читать ключевые параметры

`model_decay_lambda`

- скорость затухания старых событий;
- чем выше значение, тем быстрее модель забывает прошлые сигналы.

`model_throttle_threshold`

- порог мягкой эскалации;
- слишком низкое значение увеличит риск ложных ограничений.

`model_drop_threshold`

- порог жёсткой эскалации;
- слишком низкое значение приведёт к агрессивным блокировкам.

`model_weight_429`

- вес сигнала давления по скорости;
- подходит для постепенного накопления `score`.

`model_weight_403` и `model_weight_444`

- более сильные сигналы явно нежелательного поведения.

`model_emergency_rps`, `model_emergency_unique_ips`, `model_emergency_per_ip_rps`

- аварийные пороги на случай, когда ждать накопления `score` уже опасно.

## 16. Практический смысл для оператора

Модель следует воспринимать как автоматический локальный слой эскалации:

- базовая защита держится на L4 и L7;
- модель подхватывает аномальное поведение;
- ревизия задаёт правила работы;
- runtime применяет временные меры, пока продолжается атака.

Это не непрозрачная нейросеть, а детерминированная модель с понятными порогами, весами и временем удержания.

## 17. Что модель не делает

Модель не:

- заменяет тюнинг WAF;
- анализирует полезную нагрузку на уровне смысла;
- принимает операторские решения за человека;
- изменяет состояние control-plane сама по себе;
- переписывает активную ревизию.

Она только:

- читает телеметрию runtime;
- пересчитывает оценки IP;
- публикует адаптивные firewall-правила.

## 18. Мониторинг и отладка

Для наблюдения и диагностики используйте:

- `/healthz` и `/healthcheck`;
- раздел `Ревизии` и ленту публикаций;
- `Активность` и журналы;
- события безопасности и архив запросов;
- `adaptive.json`, если нужен детальный разбор на уровне runtime.

Признаки корректной работы:

- `adaptive.json` обновляется регулярно;
- новые проблемные IP получают стадии `throttle` или `drop`;
- после снижения активности и истечения удержания записи исчезают;
- число ложных блокировок не растёт лавинообразно.

## 19. Рекомендации по внедрению

1. Сначала включите базовую L4-защиту и мягкое L7-ограничение.
2. Затем оставьте параметры адаптивной модели консервативными.
3. Понаблюдайте реальные `429`, `403`, `444` и аварийные всплески.
4. Только после этого снижайте пороги или усиливайте веса.
