# Runbook TARINIO 2.0.0

Wiki baseline: `2.0.0`

Этот runbook описывает ежедневные операторские действия, change flow, triage и rollback для TARINIO.

## Быстрые проверки здоровья

- `GET /healthz`
- `GET /api/app/meta`
- вход через `/login`
- прохождение `/healthcheck`
- наличие ожидаемых данных на `Dashboard`
- состояние `Ревизий`

## Ежедневный операторский минимум

1. Проверить, что `/healthz` healthy.
2. Убедиться, что UI открывается без деградации и healthcheck не показывает критических ошибок.
3. Проверить в `Ревизиях`, что последняя рабочая ревизия применена успешно.
4. Сверить `Dashboard` и `События` на предмет всплеска атак, `403`, `429`, `5xx`.
5. Проверить `TLS` на предмет сертификатов с риском скорого истечения.
6. Проверить `Активность` при расследовании неожиданных изменений.

## Стандартный change flow

1. Внести изменение через `Сайты`, `TLS`, `Anti-DDoS`, `OWASP CRS` или API.
2. Проверить, что изменение соответствует ожидаемой модели.
3. Выполнить compile.
4. Применить ревизию.
5. Проверить `Dashboard`, `Запросы`, `События` и `Баны`.
6. Зафиксировать результат в change log команды.

## Где смотреть разные типы проблем

- Проблема доступности приложения: `Dashboard`, `Запросы`, `TLS`, `Сайты`
- Подозрение на ложные блокировки: `События`, `Запросы`, `Баны`, `Сайты`
- Подозрение на сетевую атаку: `Dashboard`, `Anti-DDoS`, `События`, `Баны`
- Неясно, кто менял настройки: `Активность`
- Ошибка после deploy или upgrade: `/healthz`, `healthcheck`, `Ревизии`, контейнерные логи

## Triage инцидента

### 1. Определить уровень проблемы

- login/control-plane;
- TLS и сертификаты;
- routing/upstream;
- WAF/CRS;
- rate-limit и bans;
- L4/L7 Anti-DDoS;
- runtime/container health.

### 2. Проверить последние изменения

- `Активность`
- `Ревизии`
- `Настройки`

### 3. Проверить последствия на данных

- `Dashboard`
- `Запросы`
- `События`
- `Баны`

### 4. Принять решение

- исправить настройку;
- выполнить rollback;
- временно ослабить политику;
- эскалировать инцидент.

## Rollback playbook

1. Открыть раздел `Ревизии`.
2. Найти последнюю known-good ревизию нужного сервиса.
3. Выполнить `Apply`.
4. Дождаться статуса применения.
5. Проверить восстановление на `Dashboard` и в `Запросах`.
6. Убедиться, что пользовательский трафик восстановлен.

Если rollback невозможен на уровне ревизии, переходите к rollback развёртывания и, при необходимости, к восстановлению из backup.

## Работа с банами

Если нужно срочно изолировать источник атаки:

1. Открыть `Баны`.
2. Создать ручной ban по IP и сайту.
3. При необходимости продлить ban.
4. Если IP ошибочно заблокирован, выполнить unban и проверить allowlist/denylist политику сайта.

## Работа с TLS-инцидентами

1. Проверить наличие TLS binding для сайта.
2. Проверить статус сертификата и срок действия.
3. При ACME-проблеме проверить challenge-режим и DNS/env параметры.
4. При необходимости вручную перевыпустить или импортировать сертификат.
5. После исправления перепроверить ingress и целевой host.

## Когда эскалировать

Эскалация обязательна, если:

- `/healthz` нестабилен более 5 минут;
- не проходит onboarding или login;
- compile/apply системно падает;
- после изменения политики массово растут `4xx/5xx`;
- есть признаки ложной блокировки бизнес-критичного трафика;
- есть риск истечения production-сертификата без понятного пути восстановления.
