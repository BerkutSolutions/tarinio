# Управление секретами

Этот документ объясняет, как в TARINIO `2.0.7` устроено хранение секретов, что именно делает `Vault`, какие данные допустимо оставлять локально и что оператору важно проверить после обновления или новой установки.

## Зачем это нужно

Для production- и enterprise-сценариев важны два вопроса:

- где физически живут секреты;
- остаются ли после миграции старые небезопасные копии.

В `2.0.7` продукт уже не должен выглядеть как система, где пароли backend-ов просто лежат в `.env` и расходятся по контейнерам без контроля.

## Какие классы секретов есть в системе

### Инфраструктурные

Сюда относятся:

- `POSTGRES_PASSWORD`;
- `WAF_RUNTIME_API_TOKEN`;
- `CONTROL_PLANE_SECURITY_PEPPER`;
- bootstrap-параметры для `OpenSearch`, `ClickHouse` и `Vault`.

### Интеграционные

Это секреты внешних backend-ов, например:

- пароль `ClickHouse`;
- пароль `OpenSearch`;
- `OpenSearch API key`;
- будущие токены внешних интеграций.

### Прикладные

Это значения, которые оператор сохраняет из интерфейса, а продукт потом использует для реальной работы интеграций.

## Режимы secret provider

Подсистема логгирования поддерживает два рабочих режима.

### `encrypted_file`

Это совместимый fallback-режим:

- секрет хранится в persisted runtime settings;
- перед записью он шифруется;
- API возвращает только маску;
- UI не получает plaintext обратно.

Этот режим лучше, чем хранить секреты в открытом виде, но для production он считается запасным, а не основным.

### `vault`

Это основной режим по умолчанию для `2.0.7`:

- секреты backend-интеграций пишутся в `HashiCorp Vault`;
- runtime читает их в момент использования;
- в API и UI наружу возвращаются только маски;
- локальные settings больше не должны держать рабочие backend credentials в plaintext.

## Что именно хранится в Vault

Для логгирования используются пути:

- `secret/<path_prefix>/logging/clickhouse`
- `secret/<path_prefix>/logging/opensearch`

Типичные поля:

- `password`
- `api_key`

Если в будущем появятся новые backend-ы или интеграции, они должны добавляться в тот же управляемый контур, а не храниться отдельно “временно”.

## Что остаётся локально

Даже при использовании `Vault` часть bootstrap-данных всё равно существует локально, иначе standalone-стек просто не сможет стартовать.

Допустимый минимум:

- адрес Vault;
- mount;
- path prefix;
- bootstrap token Vault;
- служебные параметры, необходимые для чтения секрета.

Недопустимо оставлять локально постоянные рабочие пароли интеграций после успешной записи в Vault.

## Как работает bootstrap в default profile

В `deploy/compose/default` Vault теперь не просто добавлен в compose, а действительно готовится к работе автоматически.

Штатный сценарий такой:

1. контейнер `vault` стартует;
2. Vault инициализируется;
3. Vault unseal-ится;
4. включается `kv-v2` mount `secret/`;
5. bootstrap token сохраняется в отдельный secret-volume;
6. `control-plane` и `runtime` получают доступ к этому token file только на чтение.

Это важно, потому что теперь standalone-профиль действительно работоспособен “из коробки”, а не требует ручного init/unseal после первого запуска.

## Порядок безопасной миграции

Безопасный порядок всегда такой:

1. проверяется доступ к Vault;
2. секрет записывается в Vault;
3. проверяется, что он читается обратно;
4. локальная persisted-копия backend credentials очищается;
5. API продолжает отдавать только маскированные значения.

Именно такой порядок нужен, чтобы не получить полусломанное состояние, когда старая копия уже удалена, а новая ещё не работает.

## Что считается небезопасным

Небезопасным считается:

- хранить рабочий backend-пароль в `.env` как постоянный источник истины;
- возвращать plaintext секрета в UI;
- оставлять старые локальные копии после успешной миграции в Vault;
- держать несколько расходящихся источников истины для одного и того же секрета.

## Что считать нормой в `2.0.7`

Штатная модель релиза такая:

- основной provider секретов: `Vault`;
- локальный `encrypted_file`: только fallback;
- backend credentials: в Vault;
- UI: только маскированные значения;
- bootstrap token: отдельный локальный технический артефакт для запуска standalone-стека.

## Как оператору проверить, что Vault действительно работает

После запуска или обновления полезно проверить:

1. `vault` находится в healthy state;
2. Vault `Initialized=true` и `Sealed=false`;
3. продукт сохраняет secret provider = `vault`;
4. после сохранения в UI backend-пароль не возвращается в plaintext;
5. runtime реально может читать секрет из Vault и использовать backend;
6. в persisted runtime settings нет старой открытой копии секрета.

## Что делать при сбое

Если Vault недоступен:

- продукт не должен мгновенно терять уже сохранённые настройки интерфейса;
- но запись новых секретов и безопасная работа интеграций могут деградировать;
- оператору нужно сначала восстановить `Vault`, а затем проверять доступность backend-ов.

Если нужно временно жить без Vault:

- допустим fallback на `encrypted_file`;
- но это следует считать временным operational mode, а не целевой production-моделью.

## Практический вывод

С точки зрения пользователя и оператора важно помнить простое правило:

- секреты интеграций должны жить в `Vault`;
- локально должны оставаться только минимальные bootstrap-данные;
- продукт не должен показывать секреты в открытом виде;
- миграция должна идти по схеме `write -> validate -> clean local copy`.

Если все эти условия выполняются, контур секретов в релизе `2.0.7` можно считать актуальным, безопасным и предсказуемым в эксплуатации.
