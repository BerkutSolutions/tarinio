# Архитектура логгирования

Этот документ описывает фактическую модель журналов запросов в TARINIO `??????? ?????`.

## Коротко

Контур запросов теперь разделён на три роли:

- `OpenSearch` — основной бэкенд для записи и чтения запросов.
- `ClickHouse` — опциональный холодный слой для корпоративного контура и долгой аналитической истории.
- локальный JSONL-архив — больше не пользовательская база запросов, а временный буфер, слой совместимости и путь восстановления.

`PostgreSQL` продолжает хранить состояние продукта: пользователей, роли, ревизии, настройки и метаданные control-plane. Это не база журналов запросов.

## Поток данных по запросам

Для каждого запроса рантайм:

1. читает журнал доступа NGINX;
2. нормализует строку в единую запись запроса;
3. пишет запись в активный бэкенд;
4. сохраняет локальную JSONL-копию как краткоживущий буфер/резерв;
5. отдаёт `GET /core-docs/api/requests` прежде всего из бэкенд-контура, а не из локального файла.

Практически это означает:

- в автономном контуре `OpenSearch` является основным источником данных для `Requests`;
- в корпоративном контуре `OpenSearch` остаётся горячим слоем, а `ClickHouse` может быть холодным источником истории;
- UI больше не должен читать архив запросов напрямую через `nginx`.

## Роль локального архива

Локальный архив остаётся в продукте, но его назначение изменилось.

Он нужен для:

- безопасного обновления старых автономных установок;
- кратковременного буфера, если бэкенд временно недоступен;
- контролируемой миграции старых `*.jsonl` архивов в `OpenSearch` или `ClickHouse`;
- диагностики и восстановления.

Он не должен оставаться основным путём выборки для страницы `Requests`.

## Обновление со старых версий

При обновлении узла со старой версии, где история запросов уже лежит в локальном архиве:

1. рантайм находит старые дневные файлы;
2. импортирует их в активный бэкенд;
3. проверяет, что ожидаемые записи реально появились в бэкенде;
4. удаляет только подтверждённый старый файл;
5. фиксирует состояние миграции.

Это делает переход `?????????? ????? -> ??????? ?????` безопасным:

- старая история не удаляется до проверки;
- автономный контур переходит на `OpenSearch` как основное хранилище;
- корпоративный контур сохраняет возможность миграции из горячего в холодный слой в `ClickHouse`.

## Автономный и корпоративный контуры

### Автономный контур

Штатный профиль `deploy/compose/default` состоит из:

- `PostgreSQL`
- `OpenSearch`
- `Vault`
- `runtime`
- `ui`
- `ddos-model`

В этом режиме:

- `OpenSearch` является основным бэкендом запросов;
- локальный архив остаётся только буферным/резервным слоем.

### Корпоративный контур

Профиль `deploy/compose/enterprise` добавляет:

- многонодовый control-plane
- `Redis` для координации высокой доступности
- `ClickHouse` для холодной аналитики

В этом режиме:

- `OpenSearch` остаётся горячим контуром;
- `ClickHouse` становится холодным слоем истории;
- старые локальные архивы по-прежнему безопасно импортируются и очищаются только после проверки.

## Что считать нормальным поведением

Штатная картина в `??????? ?????` такая:

1. свежие запросы появляются в `Requests` через API, а не через прямое чтение файлов;
2. `OpenSearch` принимает новый трафик;
3. старые файлы мигрируют и удаляются только после проверки;
4. если включён `ClickHouse`, старые горячие данные могут переноситься туда без потери подтверждённых записей;
5. локальный архив остаётся страховочным буферным слоем, а не основной базой запросов.
