# Архитектура логгирования

Этот документ описывает фактическую модель request-логов в TARINIO `2.0.7`.

## Коротко

Контур запросов теперь разделён на три роли:

- `OpenSearch` — основной backend для записи и чтения запросов.
- `ClickHouse` — опциональный cold tier для enterprise и долгой аналитической истории.
- локальный JSONL-архив — больше не пользовательская база запросов, а временный spool, слой совместимости и recovery-path.

`PostgreSQL` продолжает хранить состояние продукта: пользователей, роли, ревизии, настройки и metadata control-plane. Это не request-log database.

## Поток данных по запросам

Для каждого запроса runtime:

1. читает access log NGINX;
2. нормализует строку в единый request record;
3. пишет запись в активный backend;
4. сохраняет локальную JSONL-копию как краткоживущий spool/fallback;
5. отдаёт `GET /api/requests` прежде всего из backend-контура, а не из локального файла.

Практически это означает:

- в standalone `OpenSearch` является основным источником правды для `Requests`;
- в enterprise `OpenSearch` остаётся hot-слоем, а `ClickHouse` может быть cold-источником истории;
- UI больше не должен читать request archive напрямую через `nginx`.

## Роль локального архива

Локальный архив остаётся в продукте, но его назначение изменилось.

Он нужен для:

- безопасного обновления старых standalone-установок;
- кратковременного буфера, если backend временно недоступен;
- контролируемой миграции legacy `*.jsonl` архивов в `OpenSearch` или `ClickHouse`;
- диагностики и recovery.

Он не должен оставаться основным query-path для страницы `Requests`.

## Обновление со старых версий

При обновлении узла со старой версии, где история запросов уже лежит в локальном архиве:

1. runtime находит legacy day-файлы;
2. импортирует их в активный backend;
3. проверяет, что ожидаемые записи реально появились в backend;
4. удаляет только подтверждённый legacy-файл;
5. фиксирует состояние миграции.

Это делает переход `2.0.5 -> 2.0.7` безопасным:

- старая история не удаляется до проверки;
- standalone переходит на `OpenSearch` как primary storage;
- enterprise сохраняет возможность hot-to-cold миграции в `ClickHouse`.

## Default и Enterprise

### Default

Штатный профиль `deploy/compose/default` состоит из:

- `PostgreSQL`
- `OpenSearch`
- `Vault`
- `runtime`
- `ui`
- `ddos-model`

В этом режиме:

- `OpenSearch` является основным request-backend;
- локальный архив остаётся только spool/fallback-слоем.

### Enterprise

Профиль `deploy/compose/enterprise` добавляет:

- многонодовый control-plane
- `Redis` для HA-координации
- `ClickHouse` для cold analytics

В этом режиме:

- `OpenSearch` остаётся hot-path;
- `ClickHouse` становится cold history tier;
- legacy локальные архивы по-прежнему безопасно импортируются и очищаются только после валидации.

## Что считать нормальным поведением

Штатная картина в `2.0.7` такая:

1. свежие запросы появляются в `Requests` через API, а не через прямое чтение файлов;
2. `OpenSearch` принимает новый трафик;
3. старые legacy-файлы мигрируют и удаляются только после проверки;
4. если включён `ClickHouse`, старые hot-данные могут переноситься туда без потери подтверждённых записей;
5. локальный архив остаётся страховочным spool-слоем, а не основной базой запросов.
