# Политика поддержки и жизненного цикла

Эта страница относится к текущей ветке документации.

## Философия релизов

TARINIO следует модели движения вперёд:

- новый релиз становится основной поддерживаемой версией продукта;
- обновления предполагаются через `scripts/install-aio.sh`;
- апгрейды защищены бэкапами, post-upgrade проверками и безопасными миграциями данных;
- проект не ведёт длинную параллельную поддержку старых релизных веток.

## Что это значит на практике

Для операторов основное правило такое:

- использовать самый новый валидированный релиз, который вы готовы эксплуатировать;
- не планировать долгую жизнь на устаревших сборках;
- считать готовность к обновлению и откату частью обычной эксплуатации.

## Базовое допущение поддержки

Поддерживаемый сценарий предполагает:

- развёртывание через документированные Docker / Docker Compose профили;
- апгрейд через AIO installer или эквивалентную документированную последовательность;
- бэкап перед обновлением;
- отсутствие ручных изменений runtime-артефактов вне продуктовых workflow.

## Базовая безопасность апгрейда

В `2.0.2` обновления считаются безопасными потому, что:

- AIO installer делает лёгкие бэкапы до апгрейда;
- PostgreSQL-backed migrations версионированы;
- перенос legacy state выполнен non-destructive;
- можно включить строгий post-upgrade smoke validation;
- HA control-plane апгрейды можно проверять rolling-сценарием без простоя API.

## Ответственность оператора

Оператор должен:

- держать deployment artifacts синхронизированными с текущим релизом;
- запускать бэкапы и smoke checks;
- проверять запас по ресурсам перед обновлением;
- иметь known-good точку отката.
