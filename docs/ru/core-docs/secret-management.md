# Управление секретами

Этот документ объясняет, как в TARINIO `3.0.6` устроено хранение секретов, что именно делает `Vault`, какие данные допустимо оставлять локально и что оператору важно проверить после обновления или новой установки.

## Зачем это нужно

Для production- и enterprise-сценариев важны два вопроса:

- где физически живут секреты;
- остаются ли после миграции старые небезопасные копии.

В `3.0.6` продукт уже не должен выглядеть как система, где пароли бэкендов просто лежат в `.env` и расходятся по контейнерам без контроля.

## Какие классы секретов есть в системе

### Инфраструктурные

Сюда относятся:

- `POSTGRES_PASSWORD`;
- `WAF_RUNTIME_API_TOKEN`;
- `CONTROL_PLANE_SECURITY_PEPPER`;
- bootstrap-параметры для `OpenSearch`, `ClickHouse` и `Vault`.

### Интеграционные

Это секреты внешних бэкендов, например:

- пароль `ClickHouse`;
- пароль `OpenSearch`;
- `OpenSearch API key`;
- будущие токены внешних интеграций.

### Прикладные

Это значения, которые оператор сохраняет из интерфейса, а продукт потом использует для реальной работы интеграций.

## Режимы поставщика секретов

Подсистема логгирования поддерживает два рабочих режима.

### `encrypted_file`

Это совместимый резервный режим:

- секрет хранится в сохранённых настройках рантайма;
- перед записью он шифруется;
- API возвращает только маску;
- UI не получает открытое значение обратно.

Этот режим лучше, чем хранить секреты в открытом виде, но для промышленной среды он считается запасным, а не основным.

### `vault`

Это основной режим по умолчанию для `3.0.6`:

- секреты backend-интеграций пишутся в `HashiCorp Vault`;
- runtime читает их в момент использования;
- в API и UI наружу возвращаются только маски;
- локальные настройки больше не должны держать рабочие учётные данные бэкендов в открытом виде.

## Что именно хранится в Vault

Для логгирования используются пути:

- `secret/<path_prefix>/logging/clickhouse`
- `secret/<path_prefix>/logging/opensearch`

Типичные поля:

- `password`
- `api_key`

Если в будущем появятся новые бэкенды или интеграции, они должны добавляться в тот же управляемый контур, а не храниться отдельно «временно».

## Что остаётся локально

Даже при использовании `Vault` часть начальных данных всё равно существует локально, иначе автономный стек просто не сможет стартовать.

Допустимый минимум:

- адрес Vault;
- точка монтирования;
- path prefix;
- начальный токен Vault;
- служебные параметры, необходимые для чтения секрета.

Недопустимо оставлять локально постоянные рабочие пароли интеграций после успешной записи в Vault.

## Как работает bootstrap в default profile

В `deploy/compose/default` Vault теперь не просто добавлен в compose, а действительно готовится к работе автоматически.

Штатный сценарий такой:

1. контейнер `vault` стартует;
2. Vault инициализируется;
3. Vault unseal-ится;
4. включается `kv-v2` в точке монтирования `secret/`;
5. начальный токен сохраняется в отдельном томе секретов;
6. `control-plane` и `runtime` получают доступ к этому файлу токена только на чтение.

Это важно, потому что теперь автономный профиль действительно работоспособен «из коробки», а не требует ручной инициализации и распечатывания после первого запуска.

## Порядок безопасной миграции

Безопасный порядок всегда такой:

1. проверяется доступ к Vault;
2. секрет записывается в Vault;
3. проверяется, что он читается обратно;
4. локальная сохранённая копия учётных данных бэкенда очищается;
5. API продолжает отдавать только маскированные значения.

Именно такой порядок нужен, чтобы не получить полусломанное состояние, когда старая копия уже удалена, а новая ещё не работает.

## Что считается небезопасным

Небезопасным считается:

- хранить рабочий пароль бэкенда в `.env` как постоянный источник истины;
- возвращать открытое значение секрета в UI;
- оставлять старые локальные копии после успешной миграции в Vault;
- держать несколько расходящихся источников истины для одного и того же секрета.

## Что считать нормой в `3.0.6`

Штатная модель релиза такая:

- основной поставщик секретов: `Vault`;
- локальный `encrypted_file`: только резервный вариант;
- учётные данные бэкендов: в Vault;
- UI: только маскированные значения;
- начальный токен: отдельный локальный технический артефакт для запуска автономного стека.

## Как оператору проверить, что Vault действительно работает

После запуска или обновления полезно проверить:

1. `vault` находится в исправном состоянии;
2. Vault показывает `Initialized=true` и `Sealed=false`;
3. продукт сохраняет `vault` как активный поставщик секретов;
4. после сохранения в UI пароль бэкенда не возвращается в открытом виде;
5. рантайм реально может читать секрет из Vault и использовать бэкенд;
6. в сохранённых настройках рантайма нет старой открытой копии секрета.

## Что делать при сбое

Если Vault недоступен:

- продукт не должен мгновенно терять уже сохранённые настройки интерфейса;
- но запись новых секретов и безопасная работа интеграций могут деградировать;
- оператору нужно сначала восстановить `Vault`, а затем проверять доступность бэкендов.

Если нужно временно жить без Vault:

- допустим резервный режим `encrypted_file`;
- но это следует считать временным эксплуатационным режимом, а не целевой промышленной моделью.

## Практический вывод

С точки зрения пользователя и оператора важно помнить простое правило:

- секреты интеграций должны храниться в `Vault`;
- локально должны оставаться только минимальные начальные данные;
- продукт не должен показывать секреты в открытом виде;
- миграция должна идти по схеме «записать -> проверить -> очистить локальную копию».

Если все эти условия выполняются, контур секретов в релизе `3.0.6` можно считать актуальным, безопасным и предсказуемым в эксплуатации.
