# Корпоративная идентификация

Эта страница относится к текущей ветке документации.

Документ описывает фактически реализованную корпоративную модель идентификации в TARINIO `3.0.6`.

## Что реально реализовано

В TARINIO `3.0.6` доступны:

- интерактивный вход через `OIDC`
- provisioning пользователей через `SCIM 2.0`
- маппинг внешних групп на локальные роли
- процесс согласования для ревизий

Это реальные маршруты control-plane и реальные экраны интерфейса, а не декларативные настройки без рабочего поведения.

## Важная граница: LDAP / AD

В TARINIO `3.0.6` **нет** отдельного прямого сценария входа по `LDAP`-паролю в control-plane.

Поддержка `LDAP` / `Active Directory` реализована через тот путь, который обычно и используется в enterprise:

- пользователи каталога проходят интерактивную аутентификацию через внешний `OIDC`-провайдер
- пользователи каталога могут автоматически создаваться через мост `SCIM`
- группы каталога сопоставляются с ролями TARINIO через внешние правила сопоставления групп

То есть TARINIO поддерживает сопоставление групп `LDAP/AD`, когда группы каталога проецируются в:

- внешний claim групп в `OIDC`
- значения групп в `SCIM` payload

Это позволяет не встраивать отдельный LDAP-клиент в критический путь входа и не размывать границу между провайдером идентификации и control-plane.

## OIDC-вход

Интерактивный enterprise-вход работает через:

- `GET /core-docs/api/auth/providers`
- `GET /core-docs/api/auth/oidc/start`
- `GET /core-docs/api/auth/oidc/callback`

### Поток OIDC

1. Экран входа проверяет через `GET /core-docs/api/auth/providers`, включён ли корпоративный SSO.
2. Если провайдер включён, интерфейс показывает кнопку корпоративного SSO.
3. `GET /core-docs/api/auth/oidc/start` создаёт короткоживущий вызов входа и перенаправляет браузер на конечную точку авторизации провайдера.
4. `GET /core-docs/api/auth/oidc/callback` обменивает код на набор токенов, проверяет `id_token` и создаёт локальную сессию TARINIO.

### Что именно валидируется

Control-plane проверяет:

- `issuer`
- `audience` (`client_id`)
- подпись токена по `JWKS`
- `nonce`
- признак `email_verified`, если это включено политикой
- допустимый домен email, если настроен allow-list

### Маппинг групп

Настройки OIDC хранят:

- идентификаторы ролей по умолчанию
- сопоставления внешних групп с ролями
- имя claim, из которого берутся группы

Во время входа TARINIO вычисляет эффективные роли из:

- ролей по умолчанию
- ролей, сопоставленных внешним группам

Итоговый набор ролей фиксируется в локальной записи пользователя и попадает в сессию.

## SCIM-подготовка пользователей

Маршруты provisioning:

- `GET /scim/v2/ServiceProviderConfig`
- `GET /scim/v2/Users`
- `POST /scim/v2/Users`
- `GET /scim/v2/Users/{id}`
- `PUT /scim/v2/Users/{id}`
- `PATCH /scim/v2/Users/{id}`
- `DELETE /scim/v2/Users/{id}`
- `GET /scim/v2/Groups`

### Аутентификация SCIM

SCIM использует токены доступа bearer.

Токены создаются и удаляются через:

- `POST /core-docs/api/administration/enterprise/scim-tokens`
- `DELETE /core-docs/api/administration/enterprise/scim-tokens/{id}`

Сырые значения токенов показываются только в момент создания. При повторном чтении отдаются только метаданные:

- отображаемое имя
- prefix
- время создания
- время последнего использования

### Как ведут себя пользователи SCIM

Подготовленные пользователи сохраняются в общем каталоге TARINIO с полями:

- `auth_source=scim`
- метаданные внешней идентичности
- external groups
- `last_synced_at`

Повторная подготовка обновляет существующую запись пользователя, а не создаёт параллельные «теневые» аккаунты.

## Процесс согласования

Политика согласования настраивается в `Администрирование -> Enterprise` и применяется к маршрутам:

- `POST /core-docs/api/revisions/compile`
- `POST /core-docs/api/revisions/{revisionID}/approve`
- `POST /core-docs/api/revisions/{revisionID}/apply`

### Поведение

Когда политика согласования включена:

- новая ревизия получает статус `pending_approval`
- в ревизии сохраняется требуемое число согласований
- пользователи-рецензенты могут её согласовать
- применение блокируется до достижения порога

Запись согласования хранит:

- идентификатор пользователя
- username
- комментарий
- время согласования

### Самосогласование

Самосогласование можно отключить. В этом режиме пользователь, который собрал ревизию, не может засчитать согласование самому себе.

## Расширение модели пользователя

Корпоративная идентификация добавляет в запись пользователя:

- `auth_source`
- `external_id`
- `external_groups`
- `last_synced_at`

Это позволяет держать в одном каталоге:

- локальных bootstrap/local пользователей
- пользователей `OIDC`
- пользователей `SCIM`

и при этом не терять источник идентичности и контекст аудита.

## Что есть в UI

Раздел `Администрирование -> Enterprise` используется для:

- включения `OIDC`
- настройки issuer/client параметров
- задания ролей по умолчанию
- настройки сопоставления внешних групп
- включения `SCIM`
- выпуска и отзыва `SCIM`-токенов
- настройки политики согласования
- выгрузки подписанного пакета поддержки

## Рекомендуемая корпоративная схема

Для корпоративной directory-интеграции рекомендуется:

1. Оставить локальный bootstrap-доступ только как аварийный контур.
2. Подключить `OIDC`-провайдер, который сам работает с `LDAP/AD`.
3. Проецировать directory-группы в OIDC groups claim.
4. При необходимости автоматического жизненного цикла использовать тот же словарь групп через `SCIM`.
5. Сопоставить внешние группы с локальными ролями TARINIO в enterprise settings.
