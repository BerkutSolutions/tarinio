# Руководство по аварийному восстановлению

## Область

Этот документ описывает восстановление при потере узла, сервисов данных и при полном восстановлении из резервной копии.

## Приоритеты восстановления

Восстанавливать нужно в таком порядке:

1. постоянное состояние данных;
2. доступность control-plane;
3. готовность рантайма;
4. проверка входящего контура;
5. наблюдаемость и вторичные сервисы.

## Потеря одного узла control-plane

Если включена высокая доступность:

- продолжить работу через оставшуюся ноду;
- восстановить упавший узел из артефактов развёртывания;
- вернуть её в кластер с теми же PostgreSQL и Redis.

Если используется однокомпонентный режим:

- восстановить хост;
- вернуть тома;
- проверить `/healthz`, `/login` и `/healthcheck`.

## Потеря PostgreSQL

Основные действия:

- остановить небезопасную активность записи;
- восстановить PostgreSQL из последней валидной резервной копии;
- проверить схему и мигрированное состояние control-plane;
- подтвердить заведомо рабочую ревизию после восстановления.

## Потеря Redis в режиме высокой доступности

Основные действия:

- восстановить Redis;
- убедиться, что блокировки и поведение лидера снова работают корректно;
- проверить отсутствие расхождения состояний при компиляции и применении.

## Потеря Vault

Основные действия:

- восстановить Vault из резервной копии или snapshot;
- убедиться, что пути mTLS-сертификатов (`mtls/<site_id>/client_ca`, `upstream_cert`, `upstream_key`) и пути logging-секретов доступны;
- проверить, что control-plane успешно читает секреты после восстановления;
- если Vault был недоступен, проверить, что сайты с mTLS работают корректно после переподключения.

## Полная пересборка хоста

Рекомендуемый порядок:

1. вернуть артефакты развёртывания;
2. восстановить PostgreSQL;
3. восстановить Redis, если он используется;
4. восстановить Vault с mTLS-сертификатами и logging-секретами;
5. восстановить рантайм и тома сертификатов при необходимости;
6. запустить сервисы;
7. выполнить базовую проверку работоспособности.
