# Обновление и откат

Эта страница относится к текущей ветке документации.

Документ описывает рекомендованный жизненный цикл обновления и критерии отката.

## Поддерживаемый подход

Рекомендуемый путь обновления:

- `предыдущий стабильный релиз -> текущий релиз`

Для рабочей среды обновление должно проходить как контролируемое окно изменений, а не как фоновая операция.

## Обязательный цикл

1. Предварительная проверка
2. Резервное копирование
3. Обновление
4. Контрольная проверка
5. Решение об откате

## 1. Предварительная проверка

Перед обновлением:

- зафиксировать текущую и целевую версии;
- убедиться, что система здорова;
- убедиться, что нет зависших заданий применения;
- проверить свободное место и состояние томов;
- назначить ответственного за откат;
- зафиксировать окно обслуживания.

## 2. Резервное копирование

Перед обновлением обязательна полная резервная копия:

- БД;
- состояние рантайма;
- TLS-материалы сертификатов;
- `.env` и секреты.

Подробности: `docs/ru/core-docs/backups.md`.

## 3. Обновление

Рекомендуемая последовательность:

1. Обновить артефакты развёртывания.
2. Применить изменения compose-конфигурации.
3. Перезапустить сервисы.
4. Дождаться готовности.
5. Проверить завершение миграций.

## 4. Контрольная проверка

После обновления минимум нужно проверить:

- `GET /healthz`
- `/login`
- `/healthcheck`
- `GET /core-docs/api/app/meta` и ожидаемую версию
- открытие ключевых вкладок интерфейса
- компиляцию и применение новой или существующей ревизии
- доступность входящего трафика по HTTP/HTTPS

## 5. Решение об откате

Откат обязателен, если:

- состояние системы не стабилизируется;
- вход, первичная настройка или базовый интерфейс не работают;
- компиляция и применение системно падают;
- рантайм обслуживает трафик небезопасно или с заметной деградацией;
- есть сомнение в целостности состояния.

## Путь отката

1. Вернуть предыдущую версию развёртывания.
2. При необходимости восстановить резервную копию.
3. Проверить `/healthz`.
4. Применить заведомо корректную ревизию.
5. Повторить контрольные проверки.

## Что не делать

- не обновлять рабочую среду без резервной копии;
- не совмещать обновление с большой пачкой рискованных изменений политик;
- не пропускать контрольную проверку;
- не считать обновление завершённым только по факту старта контейнеров.
