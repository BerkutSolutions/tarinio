# Известные ограничения и границы продукта

Эта страница относится к текущей ветке документации.

Релизный baseline для этой ревизии документа: `3.0.5`.

## Зачем нужна эта страница

Точные ограничения повышают доверие сильнее, чем размытый маркетинг.

## Границы продукта

TARINIO — это:

- самостоятельно размещаемая платформа защиты и управления трафиком;
- WAF и Anti-DDoS, управляемые через ревизии;
- операторский продукт с разделением на рантайм и control-plane.

TARINIO — это не:

- SIEM;
- directory service;
- комплекс для тестирования безопасности приложений;
- универсальная замена любому edge-компоненту.

## Поддерживаемые хранилища данных

В текущем релизе продуктовый контур данных включает:

- `PostgreSQL` для состояния control-plane (пользователи, роли, ревизии, настройки, метаданные);
- `Redis` для координации высокой доступности (блокировки, leader-only задачи);
- `OpenSearch` как основной backend запросов/событий при включении;
- `ClickHouse` как опциональный cold-tier для долгосрочной аналитической истории.

Важные границы:

- `PostgreSQL` не является аналитической базой журналов запросов;
- локальные архивы запросов используются как совместимый spool/fallback слой, а не как основная система учёта.

## Границы синхронизации

Продукт реализует управляемые сценарии синхронизации:

- compile/apply ревизий с сериализацией операций в режиме высокой доступности;
- синхронизацию easy-профиля и policy-сущностей через workflows control-plane;
- контролируемую миграцию/синхронизацию legacy-архивов запросов в `OpenSearch` и при необходимости в `ClickHouse`;
- опциональный hot-to-cold поток истории запросов (`OpenSearch` -> `ClickHouse`) в enterprise-контуре.

Продукт не реализует:

- оркестрацию глобальной cross-region active-active репликации;
- ответственность за внешние SIEM-пайплайны вне документированных интеграций;
- синхронизацию с произвольными сторонними БД вне документированных backend-потоков.

## Эксплуатационные границы

Оператору всё равно нужны:

- корректное сетевое проектирование;
- ответственность за резервное копирование и аварийное восстановление;
- безопасный код приложений;
- управление секретами;
- мониторинг и процедуры реагирования на инциденты.

Дополнительно зона ответственности оператора:

- sizing и retention-политики для `OpenSearch` и `ClickHouse`;
- проверка backup/restore для каждого включённого backend;
- контроль состояния синхронизации после апгрейдов и смены топологии.
