# Доказательная база и релизы

Эта страница относится к текущей ветке документации.

Документ описывает механизмы доказательного уровня в TARINIO `??????? ?????`: цепочку аудита, подписи ревизий, подписанный пакет поддержки и релизные артефакты.

## Что реализовано

В TARINIO `??????? ?????` реализованы:

- защищённая от незаметного изменения цепочка аудита
- подпись manifest-полей ревизии
- подписанный пакет поддержки
- релизные артефакты с `checksums`, `SBOM`, `provenance` и подписью

## Защищённая цепочка аудита

Каждое audit-событие хранит:

- `prev_hash`
- `hash`

Хэш считается по нормализованному payload события и предыдущему хэшу. Поэтому изменение записи в середине цепочки ломает верификацию.

Цепочка используется не только внутри control-plane, но и выносится в материалы поддержки.

## Подпись ревизий

Во время компиляции ревизии TARINIO подписывает поля, идентифицирующие ревизию:

- ID ревизии
- версию
- время создания
- контрольную сумму
- идентичность компилятора
- состояние политики согласования

В ревизии сохраняются:

- `signature`
- `signature_key_id`

Эти данные возвращаются в каталоге ревизий и попадают в материалы поддержки.

## Связь изменение -> ревизия -> внедрение -> результат

Enterprise-контур связывает:

- создание изменения
- компиляцию ревизии
- согласование
- применение
- откат
- итоговый результат

Эта связь видна через:

- `GET /core-docs/api/revisions`
- `GET /core-docs/api/audit`
- `GET /core-docs/api/events`

Каталог ревизий показывает:

- кто собрал ревизию
- состояние согласования и список согласований
- последний результат применения
- время последнего применения
- последнее событие внедрения

## Подписанный пакет поддержки

Материалы поддержки выгружаются через:

- `GET /core-docs/api/administration/support-bundle`

Архив содержит:

- `manifest.json`
- `signature.json`
- `evidence/public_key.pem`
- `audits.json`
- `events.json`
- `jobs.json`
- `revisions.json`

### Что есть в манифесте

`manifest.json` включает:

- время генерации
- ID evidence-ключа
- список файлов
- `sha256` для каждого файла
- сводку по цепочке аудита
- количество подписанных ревизий

`signature.json` подписывает `manifest.json` ключом `Ed25519`, который хранится в хранилище корпоративных настроек.

### Как проверять пакет поддержки

1. Распакуйте архив.
2. Сверьте `sha256` файлов с `manifest.json`.
3. Проверьте подпись `Ed25519` из `signature.json` с помощью `evidence/public_key.pem`.
4. Пересчитайте цепочку аудита по `audits.json`.
5. При необходимости отдельно проверьте подписи ревизий из `revisions.json`.

## Релизные артефакты

Артефакты релиза генерируются через:

- `scripts/generate-release-artifacts.ps1`
- `scripts/release.ps1`
- `go run ./cmd/release-artifacts`

Файлы складываются в:

- `build/release/<version>/`

Набор включает:

- `release-manifest.json`
- `signature.json`
- `checksums.txt`
- `sbom.cdx.json`
- `provenance.json`
- `release-public-key.pem`

### Релизный манифест

`release-manifest.json` хранит:

- версию продукта
- git tag
- commit SHA
- docker tags
- дайджесты исходных релизных входов
- digests сгенерированных файлов
- идентификатор ключа подписи

### SBOM

`sbom.cdx.json` — это генерируемый `CycloneDX 1.5` bill of materials, собираемый из:

- `go.mod`
- `package.json`
- `package-lock.json`

Он нужен как машинно-читаемый dependency inventory конкретного релиза.

### Provenance

`provenance.json` фиксирует:

- версию релиза
- tag и commit
- идентификатор builder-а
- source inputs
- контекст генерации артефактов

Это лёгкая аттестация релиза для контура продвижения и аудиторской проверки.

## Ключи подписи

Для релизных артефактов используется отдельный постоянный ключ `Ed25519` в:

- `.work/release-signing/`

Для подписывания пакета поддержки используется корпоративный ключ доказательной базы из состояния корпоративных настроек.

Это разные контуры:

- ключ доказательной базы продукта — операторские материалы поддержки и аудита
- ключ подписи релизов — доказательства сборки и продвижения

## Практический порядок работы

Для продвижения в предпрод и прод:

1. Сгенерируйте release artifacts.
2. Архивируйте `release-manifest.json`, `signature.json`, `sbom.cdx.json` и `provenance.json` вместе с change ticket.
3. Продвигайте только те артефакты, у которых проверены контрольная сумма и подпись.
4. После внедрения выгружайте подписанный пакет поддержки для рискованных изменений.
