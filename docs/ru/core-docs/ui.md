# Интерфейс и рабочие сценарии

Эта страница относится к текущей ветке документации.

Документ описывает административный интерфейс `TARINIO`: вход, первичную настройку, основные разделы и типовые ежедневные сценарии оператора.

## Карта интерфейса

В защищенной части интерфейса доступны разделы:

- `Dashboard`
- `Sites`
- `Anti-DDoS`
- `OWASP CRS`
- `TLS`
- `Requests`
- `Revisions`
- `Events`
- `Bans`
- `Administration`
- `Activity`
- `Settings`
- `Profile`

Отдельные маршруты:

- `/login`
- `/login/2fa`
- `/onboarding/user-creation`
- `/onboarding/site-tls`
- `/onboarding/confirm`
- `/healthcheck`

## Вход и первичная настройка

### `/login`

Экран входа поддерживает:

- вход по имени пользователя и паролю;
- вход по `passkey`;
- переход на второй фактор при включенной `2FA`;
- переход в первичную настройку, если система еще не инициализирована.

### `/login/2fa`

Экран второго фактора позволяет:

- ввести код `TOTP`;
- завершить вход через `passkey`;
- вернуться на основной экран входа.

### `/onboarding/*`

Первичная настройка разбита на три шага:

1. Создание первого администратора через `POST /core-docs/api/auth/bootstrap`.
2. Создание первого сайта и `TLS`-связки.
3. Подтверждение параметров и переход в рабочий интерфейс.

## Проверка состояния

Маршрут `/healthcheck` используется как предвходная проверка состояния системы.

Проверяются:

- наличие действующей сессии;
- совместимость интерфейса и рантайма;
- доступность базовых `API`;
- целостность пользователей, ролей и административных разрешений;
- состояние модулей совместимости;
- ошибки контейнеров и рантайма.

Если проблема допускает автоматическое исправление, интерфейс может вызвать `POST /core-docs/api/app/compat/fix`.

## Панель мониторинга

`Dashboard` предназначен для ежедневного обзора состояния системы.

Раздел позволяет:

- видеть сводную статистику по трафику и атакам;
- анализировать распределение по сайтам, `URL`, `IP`, странам и кодам ответа;
- следить за загрузкой `CPU` и памяти;
- смотреть состояние контейнеров и их журналы;
- настраивать состав виджетов под роль оператора.

Позиции и размеры виджетов сохраняются при перемещении и изменении размера, а также дополнительно при `pagehide`.

## Сайты

`/services` — основной рабочий экран для настройки защищаемых сайтов.

В карточке сайта доступны вкладки:

- `Front`
- `Upstream`
- `HTTP`
- `Headers`
- `Traffic`
- `Blocking`
- `Antibot`
- `Geo`
- `ModSecurity`

Также поддерживается выбор профиля сервиса:

- `strict`
- `balanced`
- `compat`
- `api`
- `public-edge`

Профиль задает стартовый набор защитных настроек, после чего можно вручную донастроить отдельные поля.

## Логика трафика и блокировок

Для входящего трафика применяется последовательность:

1. Проверка по жестким правилам доступа (`allowlist`/`denylist`, гео-ограничения, явные черные списки).
2. При совпадении с блокирующим правилом — немедленный ответ `403`.
3. Если трафик проходит первичный фильтр, применяются проверки `Anti-Bot` и `Anti-DDoS`.

В `Traffic` настраиваются:

- списки разрешения и запрета по `IP`/подсетям;
- исключения;
- лимиты соединений и запросов;
- быстрая блокировка по типовым сигнатурам.

В `Geo` настраиваются:

- белый список стран;
- черный список стран.

Если страна находится в черном списке или не входит в белый список (когда белый список задан), запрос получает `403`.

## Anti-Bot

Во вкладке `Antibot` доступны:

- выбор режима проверки (`no`, `cookie`, `javascript`, `recaptcha`, `hcaptcha`, `turnstile`);
- адрес страницы проверки;
- правила проверки для отдельных `URL`;
- эскалация проверки;
- отдельный переключатель авто-блокировки сканеров.

Переключатель авто-блокировки сканеров включен по умолчанию и сразу возвращает `403` при явных сканерных паттернах.

## Anti-DDoS

Раздел управляет глобальными настройками `Anti-DDoS` через `API`:

- блок настроек `L4`;
- блок настроек `L7`;
- параметры адаптивной модели;
- журнал решений модели.

Типовой сценарий: сначала настраивается базовая защита, затем выполняется наблюдение по журналам и донастройка порогов.

## Запросы, события и баны

`Requests` используется для построчного анализа запросов с фильтрами и поиском.

`Events` показывает события защиты и причины срабатываний.

`Bans` объединяет активные и завершенные блокировки, а также ручные действия оператора по `IP`.

## Ревизии

Раздел `Revisions` — центральная точка применения изменений.

Поддерживается:

- компиляция изменений;
- применение ревизии;
- просмотр статусов и журналов применения;
- удаление неактивных ревизий.

## Администрирование, активность, настройки, профиль

`Administration` включает управление пользователями, ролями и связанными административными сценариями.

`Activity` показывает аудит действий операторов.

`Settings` содержит общие, системные и защитные параметры платформы.

`Profile` объединяет персональные параметры, пароль, `2FA` и `passkeys`.

## Рекомендуемый рабочий маршрут

1. Проверить `Dashboard` и `/healthcheck`.
2. При инциденте перейти в `Events`, `Requests` и `Bans`.
3. Внести изменения в `Sites`, `Anti-DDoS`, `OWASP CRS` или `TLS`.
4. Скомпилировать и применить ревизию.
5. Подтвердить результат по мониторингу и событиям.
