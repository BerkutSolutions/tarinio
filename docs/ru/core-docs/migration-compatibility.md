# Контракт миграции и совместимости

Эта страница относится к текущей ветке документации.

Релизный baseline для этой ревизии документа: `3.0.5`.

Документ фиксирует правила совместимости для логирования, миграции секретов и перехода между standalone и enterprise-контурами.

## Обратная совместимость

Существующие standalone-установки остаются поддерживаемыми:

- старые runtime-настройки нормализуются в текущую модель;
- отсутствие опциональных backend-компонентов не должно ломать старт;
- legacy-архивы запросов читаются до завершения миграции;
- очистка выполняется только после валидации целевого backend.

Ключевое правило: обновление не должно оставлять систему в частично рабочем состоянии.

## Правила миграции архива запросов

Для данных запросов порядок такой:

1. сохранить локальный legacy-архив;
2. импортировать legacy-файлы в активный backend;
3. проверить фактическое наличие данных в backend;
4. удалить только подтвержденный legacy-файл;
5. сохранить состояние миграции.

Это распространяется на:

- переходы `legacy release -> current release` в standalone;
- enterprise-сценарии hot-to-cold переноса истории (`OpenSearch -> ClickHouse`).

## Правила переключения backend

Фактическая модель в `current release`:

1. `OpenSearch` — основной backend запросов при включении;
2. `ClickHouse` — опциональный cold-tier для долгой аналитики;
3. локальный архив — spool/fallback слой, а не основной источник для UI;
4. при частичной конфигурации backend продукт должен безопасно откатываться в fallback-путь.

## Правила миграции секретов

Для logging/secret migration:

1. оператор включает `vault`;
2. TARINIO проверяет доступ к Vault;
3. backend-секреты записываются в Vault;
4. выполняется проверка обратного чтения;
5. старые локальные копии очищаются.

Инвариант: сначала запись и валидация, потом очистка legacy-копии.

## Default и Enterprise профиль

### Default

Штатный профиль `current release`:

- `PostgreSQL`
- `OpenSearch`
- `Vault`
- без обязательного `Redis`
- без обязательного `ClickHouse`

### Enterprise

Enterprise-контур добавляет:

- multi-node control-plane;
- `Redis` для HA-координации;
- `ClickHouse` для холодной аналитики.

## Что считается успешной миграцией

Миграция успешна только при одновременном выполнении:

1. стек стартует без ручной правки compose/k8s ресурсов;
2. свежие запросы видны через API/UI;
3. legacy-файлы мигрируются до удаления;
4. backend-секреты не остаются в небезопасных локальных копиях после Vault-migration;
5. standalone сохраняет fallback-путь;
6. enterprise сохраняет корректный hot-to-cold перенос.

## Что считается регрессией

Регрессия:

- удаление legacy-архивов до валидации backend;
- отображение backend как активного при фактической недоступности;
- зависимость UI от локальных файлов вместо backend API;
- очистка секретов до подтверждения сохранения в Vault;
- необходимость enterprise-зависимостей для работоспособности standalone.

## Примечание по Kubernetes/Terraform Lab

Для профиля `deploy/lab-k8s-terraform` миграционный контур считается валидным только при успешных:

1. `terraform apply` и `terraform destroy` без ручной очистки;
2. smoke-проверках single-node и HA overlay;
3. no-trace проверках без остаточных Terraform-артефактов после теста.
