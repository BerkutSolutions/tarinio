# Контракт миграции и совместимости

Этот документ фиксирует правила совместимости `??????? ?????` для логгирования, миграции секретов и перехода между standalone и enterprise-контурами.

## Обратная совместимость

Существующие автономные установки продолжают поддерживаться:

- старые настройки рантайма нормализуются в текущую модель;
- отсутствие опциональных бэкендов не ломает старт;
- архив старых запросов остаётся доступным до завершения миграции;
- очистка выполняется только после подтверждения целевого бэкенда.

Главное правило простое: обновление не должно оставлять систему в полу-рабочем состоянии.

## Правила миграции архива запросов

Для данных запросов порядок такой:

1. локальный legacy-архив сохраняется;
2. старые дневные файлы импортируются в активный бэкенд;
3. рантайм проверяет, что ожидаемые записи реально присутствуют в бэкенде;
4. только после этого удаляется подтверждённый legacy-файл;
5. состояние миграции сохраняется отдельно.

Это относится к:

- обновлению `?????????? ????? -> ??????? ?????`, где история запросов ещё жила в локальных `*.jsonl`;
- корпоративным сценариям, где подтверждённые данные дополнительно перемещаются из горячего в холодный слой.

## Правила переключения для бэкендов

Фактическая request-модель в `??????? ?????` такая:

1. `OpenSearch` — основной бэкенд для запросов, если он включён;
2. `ClickHouse` — опциональный холодный слой для корпоративной истории;
3. локальный архив остаётся буфером/резервным слоем, а не основным источником чтения для UI;
4. если бэкенд настроен неполно, продукт безопасно откатывается в резервный режим, а не делает вид, что бэкенд активен.

## Правила миграции секретов

Для секретов логгирования порядок такой:

1. оператор включает `vault`;
2. TARINIO проверяет подключение к Vault и доступ по токену;
3. backend-секреты записываются в Vault;
4. продукт проверяет, что секрет читается обратно;
5. legacy локальные копии очищаются.

Инвариант:

- сначала запись и проверка;
- потом очистка старой копии.

## Default и Enterprise

### Default

Штатный профиль `??????? ????? default` состоит из:

- `PostgreSQL`
- `OpenSearch`
- `Vault`
- без обязательного `Redis`
- без обязательного `ClickHouse`

### Enterprise

Корпоративный профиль добавляет:

- многонодовый control-plane
- `Redis` для координации высокой доступности
- `ClickHouse` для холодной аналитики

Обновление должно оставаться безопасным в обоих режимах.

## Что считать успешной миграцией

Миграция считается успешной только если одновременно выполняется всё:

1. стек поднимается без ручной правки compose;
2. свежие запросы видны через API/UI;
3. старые файлы запросов мигрируются до удаления;
4. backend-секреты больше не остаются в небезопасных локальных копиях после миграции в Vault;
5. автономная установка сохраняет безопасный буферный/резервный путь на случай временной недоступности бэкенда;
6. корпоративный профиль сохраняет проверенный перенос истории из горячего в холодный слой.

## Что считать регрессией

Регрессией считаются:

- удаление архива старых запросов до подтверждения данных в бэкенде;
- состояние, где бэкенд показан как активный, но рантайм реально не может его использовать;
- зависимость UI от локальных файлов запросов вместо API/бэкенд-пути;
- очистка локальных секретов до подтверждённой записи в Vault;
- ситуация, где автономная установка перестаёт работать без зависимостей корпоративного уровня.
