# Архитектура И Принципы Anti-DDoS Модели

Этот документ подробно описывает, как в TARINIO работает anti-DDoS контур, из каких слоёв он состоит, какие сигналы использует и как принимает решения.

## 1. Общая идея

Anti-DDoS в TARINIO не является одной “магической кнопкой”. Это каскад из трёх независимых, но связанных уровней:

1. L4 guard
   Блокирует или ограничивает вредный TCP-трафик до того, как запрос попадёт в NGINX, TLS handshake и ModSecurity.
2. Глобальный L7 rate-limit override
   Компилирует единые ограничения на HTTP-запросы в runtime artifacts ревизии.
3. Адаптивная модель
   Анализирует runtime access log, накапливает score по IP и публикует временные throttle/drop-правила в `adaptive.json`, которые затем подхватываются L4 guard.

Эти уровни не заменяют друг друга:
- L4 guard защищает сокеты и CPU до HTTP-разбора;
- L7 override держит базовый HTTP pressure под контролем;
- adaptive-модель реагирует на поведение источников и эскалирует меры динамически.

## 2. Источник истины и путь до runtime

Источник истины:
- control-plane storage: `anti_ddos_settings.json`.

Далее настройки проходят путь:
- operator changes settings;
- compile создаёт snapshot;
- validate проверяет bundle и runtime contract;
- apply публикует ревизию как активную;
- runtime читает артефакты активной ревизии.

В bundle попадают как минимум:
- `l4guard/config.json` — базовая L4-защита;
- `ddos-model/config.json` — конфиг адаптивной модели.

Сам runtime не читает domain model напрямую. Он работает только с уже скомпилированными артефактами активной ревизии.

## 3. L4 guard: что именно делает

L4 guard запускается до старта NGINX и применяет детерминированные `iptables` правила.

Базовая логика:
- ограничение параллельных TCP-соединений на один source IP через `connlimit`;
- ограничение частоты новых TCP-соединений через `hashlimit`;
- `DROP` или `REJECT` для превышений.

Это защищает от:
- high-concurrency flood;
- high-rate connection flood;
- keep-alive abuse, когда один источник удерживает слишком много открытых сокетов.

L4 guard работает на родительской цепочке:
- `DOCKER-USER`, если она доступна и выбран docker-aware сценарий;
- иначе `INPUT`.

В docker bridge / DNAT режиме используется `destination_ip`, чтобы правила ставились на правильный post-DNAT адрес runtime container.

## 4. Глобальный L7 override

Если включён `enforce_l7_rate_limit`, компилятор добавляет глобальный HTTP rate-limit override для всех включённых сайтов, кроме management UI.

Назначение этого слоя:
- не дать одному источнику или узкому набору источников безнаказанно держать высокий HTTP pressure;
- обеспечить предсказуемый базовый потолок нагрузки ещё до adaptive-эскалации.

Основные параметры:
- `l7_requests_per_second`;
- `l7_burst`;
- `l7_status_code`.

Этот слой статичен в рамках активной ревизии. Он не “учится” и не меняется сам по себе, пока оператор не выпустит новую ревизию.

## 5. Адаптивная модель: откуда берёт данные

Адаптивная модель работает отдельным процессом `ddos-model`.

Она читает:
- runtime access log;
- текущее состояние модели из `model-state.json`;
- конфиг active revision из `ddos-model/config.json`.

На выходе модель пишет:
- `adaptive.json` с временными IP-правилами.

L4 guard читает `adaptive.json` и расширяет базовые firewall-правила временными throttle/drop мерами для конкретных IP.

## 6. Какие сигналы использует модель

Модель реагирует не на “любой шум”, а на конкретные признаки давления или abusive behavior.

Базовые сигналы по HTTP status:
- `429` — базовый вес `model_weight_429`, обычно признак rate-limit pressure;
- `403` — более сильный вес `model_weight_403`, часто отражает запрещённый доступ или отклонённый шаблон;
- `444` — ещё более сильный вес `model_weight_444`, т.к. обычно это явный плохой трафик или runtime-side early close.

Дополнительные emergency-сигналы:
- резкий global RPS выше `model_emergency_rps`;
- слишком много уникальных IP в ту же секунду выше `model_emergency_unique_ips`;
- слишком высокий RPS от одного IP выше `model_emergency_per_ip_rps`.

## 7. Как считается score

Для каждого IP модель хранит запись:
- текущий `score`;
- время последнего события;
- время последнего обновления;
- текущую stage;
- `expires_at`.

Когда приходит сигнал:
- для статуса `429/403/444` к IP добавляется соответствующий вес;
- перед добавлением применяется экспоненциальное затухание по `model_decay_lambda`;
- чем дольше IP ведёт себя спокойно, тем сильнее старый score уменьшается.

Это важно:
- модель не “запоминает навсегда” случайный всплеск;
- score естественно спадает со временем;
- эскалация происходит только при повторяемом или интенсивном сигнале.

## 8. Пороги и стадии

Модель использует две главные стадии:
- `throttle`, если `score >= model_throttle_threshold`;
- `drop`, если `score >= model_drop_threshold`.

Если score ниже throttle-порога:
- IP не попадает в adaptive firewall output.

Если score выше порога throttle:
- в `adaptive.json` появляется IP с действием `throttle`.

Если score выше порога drop:
- в `adaptive.json` появляется IP с действием `drop`.

Порог `drop` всегда должен быть выше `throttle`.

## 9. Emergency detectors

### 9.1 Botnet-like burst detector

Модель агрегирует:
- количество запросов за секунду;
- число уникальных IP в ту же секунду.

Если в одну секунду одновременно выполнены условия по общему RPS и числу уникальных IP:
- по всем IP этого всплеска повышается score;
- в экстремальном случае применяется immediate drop.

Это помогает при распределённом шуме, когда один IP сам по себе не выглядит критичным, но вся группа вместе уже создаёт DDoS-паттерн.

### 9.2 Single-source flood detector

Для каждого IP считается RPS по секундам.

Если один IP выходит за emergency per-IP threshold:
- модель сразу усиливает response;
- при очень сильном превышении может поставить immediate drop, минуя постепенный набор score.

Это помогает против одиночного флуда с высокой интенсивностью.

## 10. Почему adaptive output агрегируется по сайтам

Модель хранит score в разрезе `site + ip`, но при публикации в `adaptive.json` агрегирует результат по IP.

IP попадает в итоговый adaptive output, если:
- у него есть global marker;
- или он проявил abusive pattern минимум на двух сайтах.

Это сделано специально:
- чтобы не банить IP за единичный локальный false positive на одном сайте;
- чтобы считать cross-site шум сильным признаком автоматизированной атаки.

## 11. Что попадает в adaptive.json

Формат adaptive output:
- `updated_at`;
- `throttle_rate_per_second`;
- `throttle_burst`;
- `throttle_target`;
- `entries[]`, где для каждого IP указан `action` и `expires_at`.

Возможные действия:
- `throttle`;
- `drop`.

Дальше L4 guard преобразует эти записи в конкретные `iptables` правила.

## 12. Что делает L4 guard с adaptive entries

Если запись `drop`:
- создаётся прямое firewall-правило на IP.

Если запись `throttle`:
- создаётся hashlimit-правило с параметрами `model_throttle_rate_per_second`, `model_throttle_burst`, `model_throttle_target`.

То есть adaptive-модель сама не применяет firewall. Она только публикует решение, а enforcement выполняет L4 guard.

## 13. Время удержания и истечение

После эскалации IP получает `expires_at` по `model_hold_seconds`.

Это означает:
- даже если трафик быстро стих, защита не исчезнет мгновенно;
- у runtime есть короткое удержание, чтобы атака не “пилила” систему oscillation-циклом.

При этом после истечения удержания и падения score запись удаляется из состояния и пропадает из adaptive output.

## 14. Почему модель учитывает operator ceilings

Функция effective emergency thresholds учитывает уже настроенные operator limits:
- `rate_per_second`;
- `l7_requests_per_second`;
- `conn_limit`.

Это нужно, чтобы модель:
- не начинала emergency-эскалацию слишком рано;
- не спорила с уже заданными operator ceilings;
- оставалась согласованной с политикой, которую оператор сознательно выбрал.

## 15. Как интерпретировать параметры

`model_decay_lambda`
- скорость забывания старых событий;
- выше значение — быстрее затухание и меньше память модели.

`model_throttle_threshold`
- порог мягкой эскалации;
- слишком низкое значение даст ложные throttle.

`model_drop_threshold`
- порог жёсткой эскалации;
- слишком низкое значение даст агрессивные блокировки.

`model_weight_429`
- вес сигнала rate pressure;
- подходит для мягкого накопления score.

`model_weight_403` и `model_weight_444`
- более сильные сигналы, указывающие на явно вредное поведение.

`model_emergency_rps`, `model_emergency_unique_ips`, `model_emergency_per_ip_rps`
- страховка на случай, когда ждать накопления score уже опасно.

## 16. Практический смысл для оператора

Оператор должен воспринимать модель как слой автоматической локальной эскалации:
- baseline держится через L4 + L7;
- модель подхватывает abnormal behavior;
- ревизия задаёт правила игры;
- runtime применяет временные меры, пока идёт атака.

Это не ML в смысле “непрозрачной нейросети”. Это детерминированная scoring-модель с понятными порогами, весами и TTL.

## 17. Что не делает модель

Модель не:
- заменяет WAF policy tuning;
- не анализирует payload на уровне semantic content;
- не принимает операторские решения за человека;
- не меняет control-plane state сама по себе;
- не переписывает активную ревизию.

Она только:
- читает runtime telemetry;
- пересчитывает IP scores;
- пишет adaptive firewall output.

## 18. Мониторинг и отладка

Для наблюдения используйте:
- `/healthz` и `/healthcheck`;
- раздел `Ревизии` и timeline rollout;
- `Logs / Activity`;
- security events и archived requests;
- `adaptive.json`, если нужен low-level runtime debugging.

Признаки корректной работы:
- `adaptive.json` регулярно обновляется;
- новые abusive IP получают stage `throttle` или `drop`;
- после затухания и истечения hold записи исчезают;
- ложные блокировки не растут лавинообразно.

## 19. Рекомендации по внедрению

1. Сначала включите L4 baseline и мягкий L7 baseline.
2. Затем оставьте adaptive thresholds консервативными.
3. Наблюдайте реальные `429/403/444` и emergency bursts.
4. Только после этого снижайте пороги или усиливайте веса.
