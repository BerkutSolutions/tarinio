# Практическое Руководство По Тюнингу WAF

Документ рассчитан на оператора Stage 1, у которого уже:
- поднят стек;
- завершён onboarding;
- есть рабочий single-node WAF.

Это не теоретическое описание ModSecurity. Это практическое руководство по безопасному изменению политик.

## Как WAF встроен в архитектуру

В TARINIO WAF не настраивается прямым редактированием runtime-файлов.

Поток всегда такой:

```text
control-plane -> policy changes -> compile -> revision bundle -> apply -> runtime
```

Это означает:
- оператор меняет политики в UI/API;
- control-plane создаёт новую ревизию;
- compiler генерирует детерминированный bundle;
- runtime использует только активный compiled bundle.

Ручное редактирование:
- NGINX config;
- ModSecurity config;
- файлов активного bundle

считается нарушением архитектуры и будет перезаписано следующей ревизией.

## Режимы WAF

### Detection

Используйте, когда:
- подключаете WAF к новому сайту;
- тестируете новую политику;
- разбираете false positives.

### Prevention

Используйте, когда:
- трафик сайта уже понятен;
- базовый policy baseline проверен;
- ложные блокировки взяты под контроль.

Рекомендуемый rollout:
1. `Detection`
2. наблюдение
3. `Prevention`

## Безопасный старт

Для нового сайта:
- включить `WAFPolicy.enabled = true`;
- оставить CRS включённым;
- использовать режим `Detection`.

Дальше:
1. собрать ревизию;
2. применить ревизию;
3. посмотреть activity / events / requests;
4. только после этого переходить в `Prevention`.

## Типовые проблемы

### Ложный `403`

Что делать:
1. открыть `Logs / Activity`;
2. найти событие рядом по времени;
3. определить site, revision и изменение policy;
4. внести узкое исключение;
5. собрать новую ревизию;
6. применить и перепроверить.

Правило:
- предпочитайте точечные исключения глобальному отключению CRS.

### Сайт “сломался” после включения WAF

Безопасное восстановление:
1. временно выключить WAF для сайта;
2. собрать и применить ревизию;
3. подтвердить восстановление;
4. включить WAF снова, но в `Detection`;
5. постепенно вернуть enforcement.

### Легитимные пользователи получают `429`

Проверить:
- текущую `RateLimitPolicy`;
- anti-DDoS L7 override;
- trusted proxies и реальный client IP.

Не каждый `429` является ошибкой WAF. Часто это отдельный rate-limit или anti-DDoS слой.

### Самоблокировка оператора

Проверить:
- `AccessPolicy`;
- denylist / allowlist;
- офисные, VPN и admin source ranges.

Перед ручным баном всегда перепроверяйте:
- точный IP;
- ширину CIDR;
- не затрагивает ли правило management access.

## Рекомендуемый change flow

Для WAF, Access и Rate Limit используйте один и тот же порядок:

1. изменить политику в UI/API;
2. собрать новую ревизию;
3. применить ревизию;
4. проверить поведение сайта;
5. при деградации откатиться на last known good revision.

Полезная привычка:
- делать одно осмысленное изменение за раз;
- не объединять несвязанные изменения в одну ревизию;
- проверять результат сразу после apply.

## Чем смотреть систему без SIEM

Используйте:
- UI `Logs / Activity`;
- `GET /api/audit`;
- `GET /api/events`;
- `GET /api/requests`;
- `GET /api/reports/revisions`.

Эти источники позволяют ответить на вопросы:
- кто изменил policy;
- какая ревизия была применена;
- были ли блокировки после изменения;
- как менялось поведение системы во времени.

## Когда нужно сразу откатываться

Откат на последнюю стабильную ревизию нужен, если:
- сайт массово выдаёт ложные `403`;
- после apply деградировал критичный пользовательский поток;
- anti-DDoS или rate-limit выбивают легитимный трафик;
- `/healthz` или `/healthcheck` показывают деградацию после rollout.

После отката:
- зафиксируйте причину;
- внесите более узкое изменение;
- выпустите новую ревизию отдельно от остальных правок.
