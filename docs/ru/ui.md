# Интерфейс и сценарии работы TARINIO 2.0.1

Wiki baseline: `2.0.1`

Этот документ описывает реальный административный интерфейс TARINIO по состоянию версии `2.0.1`: вход, onboarding, healthcheck, боковое меню, каждую вкладку и практические сценарии повседневной работы.

## Карта интерфейса

В защищённой части UI доступны разделы:

- `Dashboard`
- `Сайты`
- `Anti-DDoS`
- `OWASP CRS`
- `TLS`
- `Запросы`
- `Ревизии`
- `События`
- `Баны`
- `Администрирование`
- `Активность`
- `Настройки`
- `Профиль`

Отдельными маршрутами работают:

- `/login`
- `/login/2fa`
- `/onboarding/user-creation`
- `/onboarding/site-tls`
- `/onboarding/confirm`
- `/healthcheck`

## Вход и первичная настройка

### `/login`

Экран входа поддерживает:

- логин по имени пользователя и паролю;
- вход по passkey;
- автоматический переход на `2FA`, если у пользователя включён второй фактор;
- перенаправление в onboarding, если система ещё не bootstrap-нута.

### `/login/2fa`

Экран второго фактора поддерживает:

- ввод TOTP-кода;
- завершение входа через passkey как второй фактор;
- возврат на обычный экран логина.

### `/onboarding/*`

Первичная настройка разбита на 3 шага:

1. Создание первого администратора через `POST /api/auth/bootstrap`.
2. Создание первого сайта и TLS-связки.
3. Подтверждение и переход к защищённой части интерфейса.

Onboarding сразу умеет:

- создать сайт;
- создать upstream;
- выпустить self-signed или ACME-сертификат;
- привязать сертификат к сайту;
- авторизовать администратора после завершения.

## Healthcheck

Маршрут `/healthcheck` выступает как предвходной контроль состояния системы.

Он проверяет:

- наличие авторизованной сессии;
- совместимость UI/runtime контракта;
- доступность API, которые нужны для всех вкладок;
- состояние users/roles/zero-trust через административный probe;
- статус модулей совместимости;
- ошибки контейнеров и runtime.

Проверки охватывают вкладки:

- `dashboard`
- `sites`
- `revisions`
- `anti-ddos`
- `owasp-crs`
- `tls`
- `requests`
- `events`
- `bans`
- `administration`
- `activity`
- `settings`
- `profile`

Если модуль допускает автоисправление, healthcheck может вызвать `POST /api/app/compat/fix`.

## Dashboard

Главная панель предназначена для ежедневного мониторинга.

Функции раздела:

- сводная статистика по трафику и атакам;
- детализация по сайтам, URL, IP, странам и кодам ошибок;
- блок по заблокированным атакам;
- виджеты нагрузки CPU и памяти;
- обзор контейнеров;
- просмотр логов контейнеров прямо из модального окна;
- настройка набора видимых виджетов.

Практическое назначение:

- быстрый ответ на вопрос, что происходит прямо сейчас;
- проверка последствий после изменения политик;
- первичный triage инцидента до перехода в `Запросы`, `События` или `Баны`.

## Сайты

Раздел `/services` является основным конструктором конфигурации.

В списке сайтов доступны:

- просмотр сайтов в таблице;
- переход к редактированию по строке;
- создание нового сайта;
- импорт;
- экспорт.

Поддерживаются форматы импорта и экспорта:

- JSON-снимок конфигурации;
- ENV-ориентированный экспорт;
- импорт сертификатных архивов при сценариях с TLS.

### Мастер редактирования сайта

Редактор сайта разбит на вкладки.

#### `Front`

Здесь задаются:

- `server_name` и `service_id`;
- включённость сайта;
- режим безопасности;
- включение TLS;
- тип источника сертификата;
- self-signed, imported certificate или ACME-режим;
- опции Let’s Encrypt, staging и wildcard.

#### `Upstream`

Настраиваются:

- схема, хост и порт upstream;
- reverse proxy режим;
- URL и host reverse proxy;
- keepalive и websocket;
- проброс `Host`, `X-Forwarded-For`, `X-Forwarded-Proto`, `X-Real-IP`;
- SNI для reverse proxy.

#### `HTTP`

Настраиваются:

- максимальный размер тела запроса;
- `HTTP/2`;
- `HTTP/3`;
- список разрешённых методов.

#### `Headers`

Настраиваются:

- cookie flags;
- `Referrer-Policy`;
- `Content-Security-Policy`;
- `Permissions-Policy`;
- CORS.

#### `Traffic`

Настраиваются:

- allowlist и denylist;
- список исключений по IP;
- bad behavior ban;
- лимиты соединений для HTTP/1, HTTP/2, HTTP/3;
- limit request по URL;
- DNSBL;
- пресеты быстрых списков для типовых сигнатур и URI.

#### `Blocking`

Настраиваются:

- ban escalation;
- область эскалации;
- ступени эскалации времени блокировки.

#### `Antibot`

Доступны:

- выбор challenge-механизма;
- antibot URI;
- reCAPTCHA;
- hCaptcha;
- Cloudflare Turnstile;
- базовая HTTP-аутентификация как защитный барьер.

#### `Geo`

Настраиваются:

- whitelist по странам;
- blacklist по странам;
- поиск по каталогу стран.

#### `ModSecurity`

Настраиваются:

- включение ModSecurity;
- использование CRS plugins;
- кастомная конфигурация ModSecurity;
- версия CRS;
- путь и содержимое пользовательского ruleset.

### Дополнительные особенности раздела `Сайты`

- Поиск по настройкам внутри мастера помогает быстро перейти к нужному полю.
- Импорт сертификатов и экспорт материалов доступны прямо из карточки сайта.
- При сохранении UI синхронизирует сайт, upstream, TLS config, access policy и easy site profile.
- Раздел использует `easy-site-profiles` как прикладной контракт для высокоуровневой конфигурации сайта.

## Anti-DDoS

Раздел управляет глобальными настройками Anti-DDoS через `/api/anti-ddos/settings`.

Содержимое раздела:

- блок L4-настроек;
- блок L7-настроек;
- блок параметров модели;
- таблица логов модели;
- модальное окно деталей записи;
- справочный блок по работе модели.

Операции:

- создание или обновление настроек;
- просмотр событий Anti-DDoS;
- анализ того, как модель классифицирует трафик.

Этот экран нужен не только для включения защиты, но и для калибровки модели под конкретный профиль трафика.

## OWASP CRS

Раздел управляет состоянием rule set OWASP CRS.

Поддерживаются:

- просмотр текущего статуса релиза;
- dry-run проверки обновлений;
- запуск обновления;
- включение почасовой автоматической проверки обновлений.

Экран полезен, когда нужно:

- проверить, не устарел ли CRS;
- обновить runtime ruleset контролируемо;
- оценить последствия обновления до compile/apply окна.

## TLS

Раздел объединяет весь жизненный цикл сертификатов.

### Блок `Certificates`

Поддерживает:

- создание и обновление метаданных сертификатов;
- список сертификатов;
- массовый выбор;
- удаление;
- импорт архива материалов;
- экспорт выбранных материалов.

### Блок `Bindings`

Поддерживает:

- привязку `site_id -> certificate_id`;
- обновление привязки;
- удаление TLS-конфига.

### Блок `Auto Renew`

Поддерживает:

- включение автоматического продления;
- настройку `renew_before_days`.

### Блок `Upload`

Поддерживает:

- загрузку PEM-сертификата и приватного ключа;
- создание записи сертификата вместе с материалами.

### Блок `ACME`

Поддерживает:

- выпуск сертификата;
- выпуск с одновременной привязкой к сайту;
- ручное продление;
- выбор CA: `letsencrypt`, `zerossl`, `custom`;
- `http-01` и `dns-01`;
- DNS provider env;
- DNS resolvers;
- ZeroSSL EAB;
- staging и wildcard.

## Запросы

Раздел предназначен для построчного анализа request-логов.

Есть:

- фильтрация по сайту, методу, HTTP-статусу и времени;
- поиск по строке;
- пагинация;
- сортировка по столбцам;
- модальное окно с полным разбором запроса;
- raw-представление записи.

Типовые сценарии:

- найти конкретный `request_id`;
- проверить, какой URI начал получать `403` после новой политики;
- просмотреть `user-agent`, referer и upstream по проблемному трафику.

## Ревизии

Это центральный экран change-management в `2.0.1`.

Он состоит из:

- основной сетки сервисов;
- боковой колонки со статусами rollout;
- временной ленты событий ревизий;
- модального окна ревизий выбранного сервиса;
- модального окна деталей статуса.

Функции:

- просмотр ревизий по сервисам;
- просмотр активной, pending и failed ревизий;
- применение ревизии;
- удаление неактивной ревизии;
- просмотр последнего apply result и последнего события;
- очистка status timeline без удаления закреплённого результата последнего применения у самой ревизии.

Раздел работает через новый агрегированный каталог `GET /api/revisions`.

## События

Раздел предназначен для изучения security events.

Поддерживаются:

- таблица событий;
- фильтрация и поиск;
- детализация события в модальном окне;
- сопоставление события с сайтом.

Это основной экран для ответа на вопрос, какая защита сработала и почему.

## Баны

Раздел собирает и дополняет информацию по блокировкам из событий и access policy.

Поддерживаются:

- просмотр активных и истёкших банов;
- ручной бан IP по сайту;
- продление бана на стандартные интервалы;
- ручной unban;
- просмотр деталей по IP, стране, модулю и контексту события;
- синхронизация с allowlist и denylist.

Практическое применение:

- быстро изолировать источник атаки;
- отследить, что уже заблокировано автоматически;
- не потерять контекст при ручном вмешательстве.

## Администрирование

Раздел теперь является полноценным рабочим экраном, а не заглушкой.

Он включает:

- таблицу пользователей со статусом учётной записи, ролями и временем последнего входа;
- просмотр пользователя по ЛКМ через read-only модальное окно;
- явное редактирование пользователей и ролей;
- создание новых пользователей и ролей прямо из раздела;
- редактор ролей с группировкой permission-ов;
- прежний каталог административных скриптов и скачивание архивов результата.

Операционные правила:

- встроенный `admin` всегда сохраняет полный доступ;
- базовые роли теперь включают `auditor`, `manager` и `soc`;
- доступ к вкладкам определяется серверными zero-trust permissions, а не только видимостью элементов в UI.

## Активность

Раздел `/activity` показывает административный аудит через `GET /api/audit`.

Поддерживаются:

- фильтры по действию, сайту и периоду;
- постраничный просмотр;
- таймлайн административных действий;
- анализ того, кто и когда менял конфигурацию.

Это основной экран для расследования операторских изменений.

## Настройки

Раздел разбит на вкладки:

- `General`
- `Storage`
- `About`

### `General`

Позволяет:

- переключать язык интерфейса;
- включать и отключать проверку обновлений;
- запускать ручную проверку обновлений;
- видеть deployment mode и текущую версию.

### `Storage`

Позволяет:

- задавать сроки хранения логов запросов;
- задавать сроки хранения activity, events и bans;
- просматривать storage indexes;
- удалять конкретные индексы хранения.

### `About`

Показывает:

- бренд продукта;
- версию;
- ссылки на проект и организацию.

Модель доступа:

- неадминистративные пользователи всегда сохраняют доступ к `About`;
- вкладки `General` и `Storage` требуют отдельных permission-ов и больше не открываются всем авторизованным аккаунтам автоматически.

## Профиль

Раздел пользователя объединяет личные и защитные настройки аккаунта.

Поддерживаются:

- просмотр профиля;
- смена пароля;
- просмотр и настройка `2FA`;
- включение `TOTP`;
- отключение `2FA` с подтверждением;
- регистрация passkeys;
- переименование passkeys;
- удаление passkeys;
- выбор языка;
- настройка timezone-предпочтений на уровне UI.

## Навигация и общие элементы

Во всём приложении используются общие механизмы:

- клиентская навигация без полной перезагрузки;
- profile hover card с ролями и количеством permissions;
- notification center;
- фоновые session ping-запросы;
- обработка `401`, `403` и `429`;
- i18n `ru/en`.

## Рекомендуемые рабочие сценарии

### Новый сайт

1. Создать сайт в `Сайты`.
2. Настроить upstream и базовые HTTP/TLS параметры.
3. При необходимости выпустить сертификат в `TLS`.
4. Собрать ревизию.
5. Применить ревизию.
6. Проверить `Dashboard`, `Запросы` и `События`.

### Изменение защиты

1. Обновить настройки сайта, Anti-DDoS или ModSecurity.
2. Сверить ожидаемое влияние.
3. Выполнить compile/apply.
4. Следить за `Dashboard`, `Событиями` и `Банами`.
5. При деградации выполнить rollback через `Ревизии`.

### Разбор инцидента

1. `Dashboard` для общей картины.
2. `События` для точного типа срабатывания.
3. `Запросы` для уровня request detail.
4. `Баны` для блокировок и ручного вмешательства.
5. `Активность` для понимания, не совпал ли инцидент с недавним административным изменением.
