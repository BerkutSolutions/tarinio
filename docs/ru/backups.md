# Бэкапы (RU)

Базовая версия документации: `1.1.1`

## Цель

Зафиксировать рабочий минимум по резервному копированию и восстановлению для single-node развёртывания.
Документ ориентирован на операторов Docker Compose.

## Что бэкапить

- volume с PostgreSQL (состояние системы, пользователи, политики, ревизии).
- runtime/state volumes из вашего deployment-профиля.
- volumes с TLS/сертификатами, если материал хранится локально.
- `.env` и внешние секреты (вне репозитория, в зашифрованном хранилище).

## Политика бэкапов (минимум)

- Перед каждым апгрейдом: обязательный полный бэкап.
- В штатном режиме: минимум 1 раз в сутки.
- Хранить не меньше 7 последних точек восстановления.
- Минимум одна копия должна храниться вне хоста.

## Pre-backup checklist

1. Проверить здоровье стека (`/healthz`, доступ к UI).
2. Убедиться, что нет активного инцидента и rollback в процессе.
3. Зафиксировать текущую active revision id.
4. Зафиксировать текущую версию/билд.

## Базовый workflow

1. Остановить интенсивные административные изменения в UI.
2. Снять снапшоты БД и runtime volumes.
3. Архивировать `.env` и deployment overrides.
4. Сохранить метаданные бэкапа:
   - timestamp (UTC)
   - host
   - app version
   - active revision id
   - оператор

## Проверка восстановления (обязательно)

Минимум раз в месяц прогонять restore-drill в отдельной среде:

1. Поднять ту же версию приложения.
2. Восстановить БД и нужные volumes.
3. Запустить сервисы.
4. Проверить:
   - вход в систему,
   - наличие сайтов и политик,
   - успешный compile/apply,
   - доступность ожидаемых хостов.

Если восстановление не проверяется, качество бэкапа неизвестно.

## Приоритеты восстановления

- Priority 1: состояние control-plane (БД + секреты).
- Priority 2: runtime artifacts и состояние сертификатов.
- Priority 3: повторный compile/apply и проверка трафика.
